MCP Platform 由三個協同運作的套件組成：**建構**具有 UI 產生工具的 MCP 伺服器、**連接**至訊息平台，以及**組合**多個 UI 成為單一儀表板。

## 三個套件

<CardGrid>
  <Card title="@casys/mcp-server" icon="rocket">
    用於在 Deno 上建構 MCP 伺服器的框架。中介軟體管道、OAuth2、並行控制和 `McpApp`——用於註冊工具並提供 `ui://` 資源的類別。**v0.14.0，穩定版，301 個測試。**
  </Card>
  <Card title="@casys/mcp-bridge" icon="laptop">
    將 MCP App UI 橋接到訊息平台（Telegram、LINE）。執行一個 HTTP 資源伺服器，透過 WebSocket JSON-RPC 將 HTML 頁面提供到平台 webview。**v0.2.0，原型版。**
  </Card>
  <Card title="@casys/mcp-compose" icon="list-format">
    從多個工具呼叫中收集 `_meta.ui` 結果，並將它們組合成分割窗格 HTML 儀表板。事件匯流排允許 iframe 之間相互通訊。**v0.4.0，Alpha 版。**
  </Card>
</CardGrid>

## `_meta.ui` 合約

三個套件透過一個共享的慣例連接：當工具返回結果時，它包含一個 `_meta.ui.resourceUri` 欄位。該值是一個 `ui://` URI，指向在伺服器上已註冊的 HTML 頁面。

```typescript
// 帶有 UI 資源的工具結果
{
  content: [{ type: "text", text: "Invoice INV-001 loaded" }],
  _meta: { ui: { resourceUri: "ui://invoices/invoice-viewer" } }
}
```

`mcp-server` 在該 URI 上註冊 HTML。`mcp-bridge` 解析 URI 並將 HTML 提供到 Telegram Mini App webview 中。`mcp-compose` 收集多個此類結果，並在 iframe 儀表板中並排渲染它們。

## 何時使用哪個套件

**單獨使用 `mcp-server`**：當你需要 MCP 伺服器時——任何工具建構使用情境。它不依賴 `mcp-bridge` 或 `mcp-compose`。

**新增 `mcp-bridge`**：當你的使用者透過訊息平台（Telegram、LINE）與伺服器互動，且你想在平台 webview 中開啟互動式 HTML 頁面時。

**新增 `mcp-compose`**：當 LLM 工作流程呼叫多個各自返回 UI 的工具，且你想在單一儀表板中並排顯示這些 UI 時。`mcp-compose` 可以不依賴 `mcp-bridge` 使用——它渲染成純 HTML。

**三者一起使用**：當你想要完整的管道時：具有 UI 的 MCP 工具，在訊息平台中傳遞，並具有跨 UI 事件通訊。
