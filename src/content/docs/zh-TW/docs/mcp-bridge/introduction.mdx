`@casys/mcp-bridge` 將 MCP App UI 轉換為可在訊息平台 webview 中執行的 HTTP 頁面。在目前的套件中，這表示：

1. Resource Server 從本機資產目錄或後端 `ui://` 解析器提供 HTML。
2. 伺服器將 `/bridge.js?...` 注入 HTML 回應中。
3. 注入的執行時期透過 `/bridge?session=...` 在頁面與伺服器之間傳遞 MCP JSON-RPC。

如果您的 UI 已經透過 `postMessage` 使用 MCP Apps JSON-RPC 與其宿主通訊，bridge 將維持該契約並處理周圍的 webview 傳輸。

## 套件的實際功能

- `assetDirectories` 下的本機 HTML 以 `/app/<server>/<path>` 提供。
- 後端解析的 `ui://` 資源在您的後端實作 `readResource()` 時，可透過 `/ui?uri=ui://...` 公開。
- 每個提供的 HTML 頁面都會獲得一個新的記憶體內工作階段 ID 及注入的 bridge 腳本。
- bridge 腳本本地處理 `ui/initialize` 並回報協定版本 `2025-11-25`。
- 所有其他 JSON-RPC 流量透過 WebSocket 發送到伺服器，其中 `onMessage()` 先執行，`backend.handleMessage()` 後執行。

## 重要限制

- 注入的腳本以 `/bridge.js?...` 參考，瀏覽器執行時期始終在相同來源根目錄連線到 `/bridge?session=...`。請將 bridge 掛載於來源根目錄，或新增根層級重新導向規則。
- 套件僅匯出 `WebSocketTransport`，沒有已實作的 HTTP 備援傳輸。
- 工作階段狀態僅在記憶體中。
- LINE 與 Telegram 的功能不對等。
