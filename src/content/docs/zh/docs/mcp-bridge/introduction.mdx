`@casys/mcp-bridge` 将 MCP App UI 转换为可在消息平台 webview 中运行的 HTTP 页面。在当前包中，这意味着：

1. Resource Server 从本地资源目录或后端 `ui://` 解析器提供 HTML。
2. 服务器将 `/bridge.js?...` 注入到 HTML 响应中。
3. 注入的运行时通过 `/bridge?session=...` 在页面和服务器之间传递 MCP JSON-RPC 消息。

如果你的 UI 已经通过 `postMessage` 使用 MCP Apps JSON-RPC 与宿主通信，桥接层会保持该契约并处理其周围的 webview 传输。

## 已实现的功能

  <Card title="Resource Server" icon="server">
    `startResourceServer()` 提供 `/app`、`/bridge.js`、`/bridge`、`/health`、`/session`，以及可选的 `/ui`。

  <Card title="浏览器运行时" icon="puzzle">

  <Card title="Telegram" icon="rocket">

  </Card>
  <Card title="LINE" icon="setting">
    LINE 目前是部分实现：有底层适配器和浏览器运行时分支，但没有内置的 LIFF 初始化或令牌验证。
  </Card>
</CardGrid>

## 包的实际功能

- `assetDirectories` 下的本地 HTML 在 `/app/<server>/<path>` 提供服务。
- 当你的后端实现了 `readResource()` 时，后端解析的 `ui://` 资源可通过 `/ui?uri=ui://...` 暴露。
- 每个提供的 HTML 页面获得一个新的内存会话 ID 和注入的桥接脚本。
- 桥接脚本自行处理 `ui/initialize` 并报告协议版本 `2025-11-25`。
- 所有其他 JSON-RPC 流量通过 WebSocket 发送到服务器，`onMessage()` 先运行，`backend.handleMessage()` 后运行。

## 重要限制

- 注入的脚本被引用为 `/bridge.js?...`，浏览器运行时始终连接到同源根路径的 `/bridge?session=...`。将桥接挂载在源根路径，或添加根级重写规则。
- 包仅导出 `WebSocketTransport`。没有已实现的 HTTP 回退传输。
- 会话状态仅在内存中。
- LINE 与 Telegram 的功能不对等。
