## 两个浏览器端层

`@casys/mcp-bridge` 提供两个相关的浏览器端组件：

1. `startResourceServer()` 在 `/bridge.js` 提供的注入式 `bridge.js` 运行时
2. `src/core/bridge-client.ts` 中导出的 `BridgeClient` 类

它们解决相同的问题，但实现并不完全相同。

## 注入的 `bridge.js`

Resource Server 注入形如以下格式的脚本 URL：

```text
/bridge.js?platform=<platform>&session=<sessionId>[&auth=1][&debug=1]
```

该运行时：

- 从 `document.currentScript` 读取自身的查询参数
- 在 iframe 模式下 patch `window.parent.postMessage`，在独立模式下 patch `window.postMessage`
- 本地处理 `ui/initialize`
- 本地处理 `ui/open-link`
- 通过 WebSocket 转发其他所有内容
- 触发 `mcp-bridge-ready` 事件
- 触发 `mcp-bridge-auth-error` 事件

### 运行时行为

- `ui/initialize` 返回协议版本 `2025-11-25`
- 宿主信息为 `{ name: "@casys/mcp-bridge", version: "0.2.0" }`
- 待处理的请求 ID 在 30 秒后超时
- 早期收到的通知会被缓冲，直到 `ui/initialize` 完成
- WebSocket 重连最多尝试 5 次，指数退避上限为 10 秒
- 如果 socket 已关闭或认证尚未完成，出站消息将被丢弃

## 导出的 `BridgeClient`

如果你要组合自己的平台运行时而不依赖注入的脚本，请使用导出的类。

```ts
import {
  BridgeClient,
  TelegramPlatformAdapter,
  WebSocketTransport,
} from "@casys/mcp-bridge";

const client = new BridgeClient({
  serverUrl: "https://bridge.example.com",
  sessionId: "session-id",
  platform: new TelegramPlatformAdapter(),
  transport: new WebSocketTransport(),
});

await client.start();
```

## `BridgeClientOptions`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `serverUrl` | `string` | 服务器基础 URL。`BridgeClient` 追加 `/bridge?session=<id>`。 |
| `platform` | `PlatformAdapter` | 必需。提供宿主上下文、生命周期事件和 `openLink()`。 |
| `transport` | `BridgeTransport` | 代码中必需。包导出 `WebSocketTransport`。 |
| `sessionId` | `string` | 必需。传递给 WebSocket URL。 |
| `bridgeInfo` | `{ name: string; version: string }` | 可选的宿主信息覆盖。 |
| `requestTimeoutMs` | `number` | 可选。默认为 `30000`。 |

## `BridgeClient` 行为

该类：

- 初始化平台适配器
- 连接传输层
- 拦截 `window.parent.postMessage`
- 本地处理 `ui/initialize`
- 本地处理 `ui/open-link`
- 通过传输层转发其余内容
- 将 `theme-changed` 和 `viewport-changed` 生命周期事件转换为 `ui/notifications/host-context-changed`
- 在 `teardown` 生命周期事件时派发 `ui/resource-teardown`

<Aside type="caution">
  导出的 `BridgeClient` 只 patch `window.parent.postMessage`（iframe 模式）。它不支持独立模式（当 `window.parent === window` 时 patch `window.postMessage`）。独立模式仅通过注入的 `bridge.js` 脚本提供。
</Aside>

## 与注入运行时的重要差异

- 导出的 `BridgeClient` 未实现注入运行时的重连循环。
- 导出的 `BridgeClient` 未实现注入运行时的认证消息协议。
- 导出的 `BridgeClient` 仅支持 iframe 模式（`window.parent.postMessage`）。独立模式仅限注入运行时。
- 目前包中只实现了 `WebSocketTransport`。

## `McpAppsMethod` 常量

该包导出一个 `McpAppsMethod` 对象，包含 MCP Apps 协议中所有已知的 JSON-RPC 方法名称字符串。当你需要引用方法名称而不想硬编码字符串时，可以导入它。

```ts
import { McpAppsMethod } from "@casys/mcp-bridge";
```

| 常量 | 方法字符串 | 方向 |
| --- | --- | --- |
| `UI_INITIALIZE` | `ui/initialize` | App → Host |
| `UI_NOTIFICATIONS_INITIALIZED` | `ui/notifications/initialized` | App → Host |
| `TOOLS_CALL` | `tools/call` | App → Host |
| `RESOURCES_READ` | `resources/read` | App → Host |
| `UI_OPEN_LINK` | `ui/open-link` | App → Host |
| `UI_MESSAGE` | `ui/message` | App → Host |
| `UI_UPDATE_MODEL_CONTEXT` | `ui/update-model-context` | App → Host |
| `UI_REQUEST_DISPLAY_MODE` | `ui/request-display-mode` | App → Host |
| `NOTIFICATIONS_MESSAGE` | `notifications/message` | App → Host |
| `UI_TOOL_INPUT` | `ui/notifications/tool-input` | Host → App |
| `UI_TOOL_INPUT_PARTIAL` | `ui/notifications/tool-input-partial` | Host → App |
| `UI_TOOL_RESULT` | `ui/notifications/tool-result` | Host → App |
| `UI_TOOL_CANCELLED` | `ui/notifications/tool-cancelled` | Host → App |
| `UI_HOST_CONTEXT_CHANGED` | `ui/notifications/host-context-changed` | Host → App |
| `UI_SIZE_CHANGED` | `ui/notifications/size-changed` | Host → App |
| `UI_RESOURCE_TEARDOWN` | `ui/resource-teardown` | Host → App |

`BridgeClient` 在本地处理 `UI_INITIALIZE` 和 `UI_OPEN_LINK`。所有其他 App → Host 方法通过传输层转发到 resource server。
