Every tool call in `@casys/mcp-server` flows through a composable middleware chain — the same onion model you know from Hono, Koa, or Express. This is how auth, rate limiting, validation, and your custom logic all compose together without tangling.

## How it works

<img src="/diagrams/mcp-server/mcp-middleware-pipeline.svg" alt="Middleware Pipeline" style="width: 100%; max-width: 600px; margin: 24px auto; display: block;" />

<Mermaid chart={`graph LR

  CUSTOM --> VAL[Validation]

```
Request → Rate Limit → Auth → [Custom] → Scope Check → Validation → Backpressure → Handler
                                                                                    ↓
Response ← Rate Limit ← Auth ← [Custom] ← Scope Check ← Validation ← Backpressure ← Result
```

Each middleware receives a `context` object and a `next()` function. Call `next()` to pass control downstream. The response flows back through the same chain in reverse — so a single middleware can act on both the request and the response.

## Writing a middleware

A middleware is just an async function. Here's a logger that times every tool call:

```typescript
import type { Middleware } from "@casys/mcp-server";

const logging: Middleware = async (ctx, next) => {
  const start = performance.now();
  console.log(`→ ${ctx.toolName}`, ctx.args);

  const result = await next();

  const ms = (performance.now() - start).toFixed(0);
  console.log(`← ${ctx.toolName} (${ms}ms)`);
  return result;
};
```

No base class, no decorators, no registration boilerplate.

## Registering middlewares

Use `server.use()` to add your middlewares. They slot in between the built-in auth layer and the built-in scope-check/validation layers:

```typescript
const server = new ConcurrentMCPServer({
  name: "my-server",
  version: "1.0.0",
});

server.use(logging);
server.use(metrics);
server.use(caching);

server.registerTool(/* ... */);
await server.startHttp({ port: 3000 });
```

`use()` is chainable:

```typescript
server.use(logging).use(metrics).use(caching);
```

<Aside type="caution">
  `use()` must be called **before** `start()` or `startHttp()`. The pipeline is compiled at startup and cannot be modified after.
</Aside>

## Pipeline order

Here's exactly what runs for every tool call, in order:

| Order | Layer | Source | What it does |
|-------|-------|--------|-------------|
| 1 | Rate Limit | Built-in (if configured) | Sliding-window per-client throttling |
| 2 | Auth | Built-in (if configured) | JWT/Bearer token validation, populates `ctx.authInfo` |
| 3 | **Your middlewares** | `server.use()` | Your code, in registration order |
| 4 | Scope Check | Built-in (if auth + `requiredScopes`) | Verifies token scopes match tool definition |
| 5 | Validation | Built-in (if `validateSchema: true`) | JSON Schema validation via ajv 8 |
| 6 | Backpressure | Built-in | Queue/sleep/reject based on `maxConcurrent` |
| 7 | Handler | `registerTool()` | Your tool handler function |

<Aside type="tip">
  Your middlewares run **after** auth but **before** scope checking and validation. This means `ctx.authInfo` is already populated when your middleware runs.
</Aside>

## The context object

Every middleware receives a `MiddlewareContext`:

```typescript
interface MiddlewareContext {
  toolName: string;                  // Which tool is being called
  args: Record<string, unknown>;     // The tool's input arguments
  request?: Request;                 // HTTP Request (HTTP transport only)
  sessionId?: string;                // Session ID (HTTP transport only)
  authInfo?: AuthInfo;               // Populated by auth middleware (if configured)
  [key: string]: unknown;            // Extensible — add your own fields
}
```

`authInfo` is available after the auth layer:

```typescript
interface AuthInfo {
  subject: string;                     // Token "sub" claim
  clientId?: string;                   // Token "client_id" or "azp" claim
  scopes: string[];                    // Parsed from "scope" claim
  claims?: Record<string, unknown>;    // All token claims
  expiresAt?: number;                  // Token expiry as Unix epoch seconds
}
```

## Enriching context

Middlewares can attach data for downstream middlewares and the handler:

```typescript
const tenantResolver: Middleware = async (ctx, next) => {
  const tenantId = ctx.request?.headers.get("x-tenant-id");
  if (!tenantId) {
    throw new Error("Missing X-Tenant-Id header");
  }
  ctx.tenantId = tenantId;  // Available to all downstream middlewares + handler
  return next();
};
```

## Short-circuiting

Don't call `next()` to skip the rest of the pipeline. This is how you build caching:

```typescript
const cache = new Map<string, { result: unknown; expiry: number }>();

const caching: Middleware = async (ctx, next) => {
  const key = `${ctx.toolName}:${JSON.stringify(ctx.args)}`;
  const cached = cache.get(key);

  if (cached && cached.expiry > Date.now()) {
    return cached.result;  // Entire downstream pipeline is skipped
  }

  const result = await next();
  cache.set(key, { result, expiry: Date.now() + 60_000 });
  return result;
};
```

## Error handling

Errors thrown anywhere in the chain propagate back through all middlewares. Put your error handler first so it wraps everything:

```typescript
const errorHandler: Middleware = async (ctx, next) => {
  try {
    return await next();
  } catch (error) {
    console.error(`Tool ${ctx.toolName} failed:`, error);
    // Re-throw so the framework converts it to an MCP error response.
    // Returning a plain object here produces a normal (non-error) result.
    throw error;
  }
};

// First middleware = outermost layer = catches all errors
server.use(errorHandler);
server.use(logging);
server.use(caching);
```

<Aside type="caution">
  Returning `{ error: true, message }` from a middleware serializes as a **normal** tool result, not an MCP `isError` response. To signal a tool error, throw an `Error` (or let it propagate) — the framework's `toolErrorMapper` will convert it to the correct MCP error format.
</Aside>

## Recipes

### Request timing

```typescript
const timing: Middleware = async (ctx, next) => {
  const start = performance.now();
  try {
    return await next();
  } finally {
    const duration = performance.now() - start;
    metrics.record(ctx.toolName, duration);
  }
};
```

### Tool-specific access control

```typescript
const adminOnly: Middleware = async (ctx, next) => {
  if (ctx.toolName.startsWith("admin_")) {
    const role = ctx.authInfo?.claims?.role;
    if (role !== "admin") {
      throw new Error("Admin access required");
    }
  }
  return next();
};
```

### Input sanitization

```typescript
const sanitizer: Middleware = async (ctx, next) => {
  if (typeof ctx.args.query === "string") {
    ctx.args.query = ctx.args.query.trim().slice(0, 1000);
  }

  const result = await next();

  return { ...result, processedAt: new Date().toISOString() };
};
```
