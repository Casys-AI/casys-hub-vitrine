MCP Apps let you serve interactive HTML UIs as resources alongside your tools. Clients that support the `text/html;profile=mcp-app` MIME type (MCP SDK 1.27) will render them inline — dashboards, data viewers, forms, config panels, right next to your tools.

## The MIME type

```typescript
import { MCP_APP_MIME_TYPE } from "@casys/mcp-server";
// "text/html;profile=mcp-app"
```

Use `MCP_APP_MIME_TYPE` (the exported constant) as the MIME type for any resource that should be rendered as an interactive UI.

## Registering a UI resource

```typescript
import { ConcurrentMCPServer, MCP_APP_MIME_TYPE } from "@casys/mcp-server";

const server = new ConcurrentMCPServer({
  name: "my-server",
  version: "1.0.0",
});

server.registerResource(
  {
    uri: "ui://my-server/viewer",
    name: "Data Viewer",
    description: "Interactive data viewer",
  },
  async (uri) => ({
    uri: uri.toString(),
    mimeType: MCP_APP_MIME_TYPE,
    text: `
      <html>
        <body>
          <h1>Data Viewer</h1>
          <div id="app"></div>
        </body>
      </html>
    `,
  }),
);

await server.start();
```

## Dynamic UI resources

The handler receives the URI and can return different content based on fetched data or any other state:

```typescript
server.registerResource(
  {
    uri: "ui://my-server/dashboard",
    name: "Dashboard",
    description: "Real-time metrics dashboard",
  },
  async (uri) => {
    const metrics = await getMetrics();
    return {
      uri: uri.toString(),
      mimeType: MCP_APP_MIME_TYPE,
      text: renderDashboard(metrics),  // Fresh data on every read
    };
  },
);
```

## Tool `_meta.ui` — linking tools to UI views

SDK 1.27 adds `_meta.ui` to tool definitions to describe how a tool's output connects to a UI:

```typescript
server.registerTool(
  {
    name: "get_report",
    description: "Fetch a sales report",
    inputSchema: {
      type: "object",
      properties: { reportId: { type: "string" } },
      required: ["reportId"],
    },
    _meta: {
      ui: {
        resourceUri: "ui://my-server/viewer",       // Which viewer renders this tool's output
        visibility: ["model", "app"],               // Visible to both LLM and UI
        emits: ["report:loaded"],                   // Events this tool emits to the UI
        accepts: ["filter:changed"],                // Events this tool accepts from the UI
      },
    },
  },
  async ({ reportId }) => fetchReport(reportId),
);
```

## App-only tools

Register tools visible only to UI clients, not to the LLM:

```typescript
server.registerAppOnlyTool(
  {
    name: "update_filter",
    description: "Update dashboard filter",
    inputSchema: {
      type: "object",
      properties: { filter: { type: "string" } },
    },
  },
  async ({ filter }) => applyFilter(filter),
);
```

## Bulk viewer registration

`registerViewers()` registers file-based viewer resources using a `RegisterViewersConfig` object:

```typescript
server.registerViewers({
  prefix: "ui://my-server/",
  moduleUrl: import.meta.url,
  exists: async (path) => {
    try {
      await Deno.stat(path);
      return true;
    } catch {
      return false;
    }
  },
  readFile: async (path) => await Deno.readTextFile(path),
});
```

The framework uses `exists` and `readFile` to serve viewer files under the given `prefix`. `moduleUrl` resolves relative paths to absolute filesystem paths.

## Structured tool results

Tools can return both a text response (for the LLM) and structured data (for the UI) simultaneously:

```typescript
import type { StructuredToolResult } from "@casys/mcp-server";

server.registerTool(
  {
    name: "get_data",
    description: "Fetch data",
    inputSchema: { type: "object", properties: {} },
    outputSchema: {
      type: "object",
      properties: {
        rows: { type: "array" },
        total: { type: "number" },
      },
    },
  },
  async (): Promise<StructuredToolResult> => {
    const data = await fetchData();
    return {
      content: `Found ${data.rows.length} rows`,   // LLM-readable string
      structuredContent: {                          // Machine-readable for UI
        rows: data.rows,
        total: data.rows.length,
      },
    };
  },
);
```

## Content Security Policy

When serving HTML resources, inject a CSP header to control what the iframe can load:

```typescript
import {
  buildCspHeader,
  injectCspMetaTag,
} from "@casys/mcp-server";

// Build a CSP header string
const csp = buildCspHeader({
  scriptSources: ["'self'", "'unsafe-inline'"],
  connectSources: ["'self'"],
  imgSources: ["'self'", "data:"],
  frameAncestors: ["'self'"],
  allowInline: false,
});

// Or inject a <meta> CSP tag directly into HTML
const html = `<html><head></head><body>...</body></html>`;
const secureHtml = injectCspMetaTag(html, csp);
```

You can also set CSP globally via `resourceCsp` in the server options:

```typescript
const server = new ConcurrentMCPServer({
  name: "my-server",
  version: "1.0.0",
  resourceCsp: {
    scriptSources: ["'self'"],
    connectSources: ["'self'"],
  },
});
```

## PostMessage authentication (HMAC)

To verify messages sent between the UI iframe and the host, use `MessageSigner` and `injectChannelAuth`:

```typescript
import {
  injectChannelAuth,
} from "@casys/mcp-server";

// secret must be a 64-character hex string
const secret = "a1b2c3..."; // 64 hex chars

// In your UI HTML, inject auth credentials before serving
const html = injectChannelAuth(rawHtml, secret);
```

The injected script handles HMAC signing of postMessage events, so the host can verify that messages originate from your UI.

## Checking registered resources

```typescript
server.getResourceCount();  // 2
```

<Aside>
  Not all MCP clients render `text/html;profile=mcp-app` resources. Clients without support simply won't show the UI — no errors, no broken tool calls.
</Aside>
