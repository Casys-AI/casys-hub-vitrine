## Server setup

```ts
import { JsonRpcMcpBackend, startResourceServer } from "@casys/mcp-bridge";

const server = startResourceServer({
  assetDirectories: {
    "my-app": "./ui-dist",
  },
  platform: "telegram",
  telegramBotToken: Deno.env.get("TELEGRAM_BOT_TOKEN")!,
  backend: new JsonRpcMcpBackend({
    endpointUrl: "https://my-mcp.example.com/mcp",
  }),
});
```

For `platform: "telegram"`, the server fails fast unless you provide either:

- `telegramBotToken`
- a custom `auth` handler

## What happens in the browser

When the injected runtime sees `platform=telegram`, it uses `window.Telegram.WebApp` if it is available.

It currently does all of the following:

- calls `Telegram.WebApp.ready()`
- calls `Telegram.WebApp.expand()` when the app is not already expanded
- sends `initData` over the WebSocket auth channel when auth is required
- uses `Telegram.WebApp.openLink()` for `ui/open-link`
- builds `hostContext.theme` from `colorScheme`
- builds `hostContext.containerDimensions.height` from `viewportStableHeight`
- listens to `themeChanged` and `viewportChanged`

It does not currently bridge the rest of the Telegram WebApp event surface.

<Aside type="caution">
  The injected `bridge.js` uses `height` (not `maxHeight`) in `containerDimensions`. The exported `TelegramPlatformAdapter` class uses `maxHeight` instead (see `getContainerDimensions()`). The `ContainerDimensions` type supports both fields. Code that reads host context from `ui/initialize` must check for the field name that matches the layer it is running against.
</Aside>

## Authentication

Telegram auth is implemented on the server, not in the browser adapter.

The built-in handler:

1. parses the Mini App `initData`
2. removes `hash`
3. sorts the remaining `key=value` pairs
4. derives the secret with HMAC-SHA256 using the `"WebAppData"` key
5. verifies the signature with `crypto.subtle.verify`
6. rejects `auth_date` values older than 24 hours or more than 60 seconds in the future

Auth happens on the first WebSocket message after the session already exists.

<Aside type="caution">
  Keep the bot token on the server. The client only sends Telegram-provided `initData`; signature verification happens server-side.
</Aside>

## Host context details

For the auto-injected runtime, Telegram currently contributes:

- `theme`
- `styles.variables` derived from `themeParams` (see CSS variables below)
- `locale` from `initDataUnsafe.user.language_code` when available, falling back to `navigator.language`
- `timeZone` from the browser environment
- `containerDimensions.width` from `window.innerWidth`
- `containerDimensions.height` from `viewportStableHeight` (falls back to `window.innerHeight`)

The exported `TelegramPlatformAdapter` contributes the same fields but with two differences:

- `locale` always comes from `navigator.language` — it does not read `initDataUnsafe.user.language_code`
- `containerDimensions` uses `maxHeight` (not `height`) for the viewport height value

## CSS variable naming

The two Telegram layers produce different CSS variable schemas in `hostContext.styles.variables`.

**Injected `bridge.js` runtime** — keys mirror Telegram's `themeParams` property names prefixed with `--tg-`:

| Telegram `themeParams` key | CSS variable |
| --- | --- |
| `bg_color` | `--tg-bg-color` |
| `secondary_bg_color` | `--tg-secondary-bg-color` |
| `text_color` | `--tg-text-color` |
| `button_color` | `--tg-button-color` |
| `button_text_color` | `--tg-button-text-color` |
| `hint_color` | `--tg-hint-color` |
| `link_color` | `--tg-link-color` |
| (any other `themeParams` key) | `--tg-<key-with-dashes>` |

**Exported `TelegramPlatformAdapter`** — keys use a semantic naming convention:

| Telegram `themeParams` key | CSS variable |
| --- | --- |
| `bg_color` | `--color-background-primary` |
| `secondary_bg_color` | `--color-background-secondary` |
| `text_color` | `--color-text-primary` |
| `subtitle_text_color` | `--color-text-secondary` |
| `section_separator_color` | `--color-border-primary` |
| `accent_text_color` | `--color-ring-primary` |
| `hint_color` | `--color-text-hint` |
| `link_color` | `--color-text-link` |
| `button_color` | `--color-button-primary` |
| `button_text_color` | `--color-button-text` |
| `header_bg_color` | `--color-background-header` |
| `section_bg_color` | `--color-background-section` |

If you are using the injected runtime, target `--tg-*` variables in your CSS. If you are using `TelegramPlatformAdapter` with a custom `BridgeClient`, target the semantic `--color-*` variables.

## Exported Telegram APIs

The package also exports two Telegram-specific classes:

- `TelegramAdapter`: a low-level `postMessage` adapter
- `TelegramPlatformAdapter`: a higher-level platform adapter for custom `BridgeClient` usage

`TelegramPlatformAdapter` exposes more structured host context than the injected runtime, including `safeAreaInsets`, `getAuthData()`, and `sendMessage()`. Its `sendMessage()` method calls `Telegram.WebApp.sendData()`, which closes the Mini App after sending.

## What not to assume

- Safe area insets are available from `TelegramPlatformAdapter`, but the auto-injected `bridge.js` runtime does not currently include them in the host context it sends to the app.
- `MainButton`, `BackButton`, and the long list of Telegram SDK events are typed in the source, but they are not currently wired into the bridge flow.
