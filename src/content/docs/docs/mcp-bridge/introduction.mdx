**@casys/mcp-bridge** (v0.2.0) connects MCP Apps — interactive UIs served via `ui://` resources — to messaging platforms like Telegram and LINE, without modifying your existing MCP App or MCP Server.

Available on [JSR](https://jsr.io/@casys/mcp-bridge) and [npm](https://www.npmjs.com/package/@casys/mcp-bridge). Deno-first; Node.js 20+ supported via Web Crypto API.

## The problem

MCP Apps (`ui://` resources) only run inside developer tools like Claude Desktop or VS Code. Users on Telegram or LINE cannot access them.

Bridging platforms manually means:
- Rewriting UI code for each platform SDK
- Re-implementing auth, session management, and security per platform
- Maintaining separate codebases
- Losing the MCP protocol's native tool execution model

## The fix

`@casys/mcp-bridge` inserts a transparent bridge between your MCP App and the messaging platform:

1. **Intercepts** `postMessage` calls from your MCP App via an injected IIFE (`bridge.js`)
2. **Routes** JSON-RPC messages over WebSocket to a Resource Server
3. **Forwards** tool calls to your unmodified MCP Server
4. **Maps** platform-specific auth, theme, and viewport automatically

```typescript
import { startResourceServer } from "@casys/mcp-bridge";

startResourceServer({
  assetDirectories: { "my-app": "./ui-dist" },
  platform: "telegram",
  telegramBotToken: Deno.env.get("TELEGRAM_BOT_TOKEN"),
  onMessage: async (_session, message) => {
    return backend.handleMessage(message);
  },
  options: { resourceServerPort: 4000 },
});
```

## Three-layer architecture

<CardGrid>
  <Card title="bridge.js (IIFE)" icon="puzzle">
    Injected into served HTML. Monkey-patches `postMessage`, opens WebSocket, handles auth-on-connect, auto-reconnects with exponential backoff (max 5 attempts). Handles `ui/initialize` and `ui/open-link` locally.
  </Card>
  <Card title="Resource Server" icon="server">
    `Deno.serve()` HTTP + WebSocket server. Endpoints: `GET /health`, `POST /session`, `GET /bridge.js`, `WS /bridge?session=id`, `GET /app/<server>/<path>`, `GET /ui?uri=ui://...`.
  </Card>
  <Card title="Platform Adapters" icon="setting">
    TelegramPlatformAdapter (complete): HMAC-SHA256 initData validation, theme params mapped to CSS custom properties. LineAdapter (partial): extends BasePostMessageAdapter, uses `liff.getAccessToken()`.
  </Card>
  <Card title="MCP Server" icon="rocket">
    Your existing MCP Server stays completely unmodified. The Resource Server connects to it via `JsonRpcMcpBackend` as a standard MCP client.
  </Card>
</CardGrid>

## Status

- **120 tests** covering all three layers
- **Telegram**: complete — auth, theme mapping, viewport, safe areas
- **LINE**: partial — adapter implemented, no built-in auth handler
- **Node.js 20+**: supported via Web Crypto API (no Deno.serve — use your own HTTP server)

<Aside type="caution">
  The LINE adapter is available but does not include a built-in auth handler. You must implement your own server-side LINE auth.
</Aside>
