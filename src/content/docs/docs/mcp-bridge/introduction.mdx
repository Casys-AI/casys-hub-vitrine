`@casys/mcp-bridge` turns an MCP App UI into an HTTP page that can run inside a messaging-platform webview. In the current package, that means:

1. A Resource Server serves HTML from local asset directories or a backend `ui://` resolver.
2. The server injects `/bridge.js?...` into HTML responses.
3. The injected runtime moves MCP JSON-RPC between the page and the server over `/bridge?session=...`.

If your UI already talks to its host with MCP Apps JSON-RPC over `postMessage`, the bridge keeps that contract and handles the webview transport around it.

## What is implemented

<CardGrid>
  <Card title="Resource Server" icon="server">
    `startResourceServer()` serves `/app`, `/bridge.js`, `/bridge`, `/health`, `/session`, and optionally `/ui`.
  </Card>
  <Card title="Browser Runtime" icon="puzzle">
    The injected `bridge.js` handles `ui/initialize` and `ui/open-link` locally, then forwards the rest over WebSocket.
  </Card>
  <Card title="Telegram" icon="rocket">
    Telegram is the only end-to-end platform path with built-in server-side auth based on Mini App `initData`.
  </Card>
  <Card title="LINE" icon="setting">
    LINE is partial today: there is a low-level adapter and a browser runtime branch, but no built-in LIFF initialization or token validation.
  </Card>
</CardGrid>

## What the package actually does

- Local HTML under `assetDirectories` is served at `/app/<server>/<path>`.
- Backend-resolved `ui://` resources can be exposed at `/ui?uri=ui://...` when your backend implements `readResource()`.
- Each served HTML page gets a new in-memory session ID and an injected bridge script.
- The bridge script handles `ui/initialize` itself and reports protocol version `2025-11-25`.
- All other JSON-RPC traffic goes to the server over WebSocket, where `onMessage()` runs first and `backend.handleMessage()` runs second.

## Important limits

- The injected script is referenced as `/bridge.js?...`, and the browser runtime always connects to `/bridge?session=...` at the same origin root. Mount the bridge at the origin root or add root-level rewrites.
- The package exports only a `WebSocketTransport`. There is no implemented HTTP fallback transport.
- Session state is in-memory only.
- LINE is not feature-parity with Telegram.
