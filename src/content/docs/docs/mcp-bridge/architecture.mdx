## Overview

```
[MCP App (WebView)]
  --postMessage--> [bridge.js IIFE]
  --WebSocket-->   [Resource Server (Deno.serve)]
  --JSON-RPC-->    [JsonRpcMcpBackend -> MCP Server]
```

The bridge sits between the messaging platform's WebView and your MCP Server. Neither your MCP App nor your MCP Server requires any changes.

## Layer 1: bridge.js (IIFE)

`bridge.js` is a self-executing script injected by the Resource Server into every HTML response via `injectBridgeScript()`.

**What it does on load:**
1. Monkey-patches `window.parent.postMessage` to intercept outgoing JSON-RPC messages
2. Opens a WebSocket to `WS /bridge?session=<id>` (session ID is embedded during injection)
3. Sends auth as the first WebSocket message (platform `initData` or token)
4. Handles `ui/initialize` and `ui/open-link` locally — these never reach the server
5. Forwards all other JSON-RPC messages to the Resource Server
6. Dispatches incoming WebSocket messages back as `MessageEvent` on the app window

**Reconnection:** exponential backoff, maximum 5 attempts. After 5 failures the bridge stops and logs an error.

**Client-side exports used internally:**

```typescript
import {
  BridgeClient,
  BasePostMessageAdapter,
  MessageRouter,
  WebSocketTransport,
} from "@casys/mcp-bridge";
```

## Layer 2: Resource Server

The Resource Server is a `Deno.serve()` HTTP server started by `startResourceServer()`.

### HTTP Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Returns `{ status: "ok" }`. Use for uptime checks. |
| `/session` | POST | Creates a session. Validates platform auth (Telegram: HMAC-SHA256 `initData`). Returns session ID. HTTP 429 if session limit (10,000) reached. |
| `/bridge.js` | GET | Serves the compiled bridge IIFE. Cached. |
| `/bridge` | WS | WebSocket endpoint. Requires `?session=<id>`. Session must exist before connecting. Auth as first message. |
| `/app/<server>/<path>` | GET | Serves static assets from `assetDirectories[server]`. Path traversal protected. |
| `/ui` | GET | Resolves `?uri=ui://...`, serves HTML with `bridge.js` injected and CSP header applied. |

### Session lifecycle

```
POST /session  ->  session created, TTL 30 min
WS /bridge     ->  session associated with WebSocket
                   inactivity cleanup every 60s
                   max 10,000 sessions (HTTP 429 beyond)
```

### Hooks

```typescript
startResourceServer({
  // Called for every JSON-RPC message from a WebView
  onMessage: async (session, message) => {
    return backend.handleMessage(message); // or null to ignore
  },
  // Called for raw HTTP requests (before routing)
  onHttpRequest: async (req) => {
    return null; // return a Response to short-circuit
  },
});
```

### CSP generation

`buildCspHeader()` generates a `Content-Security-Policy` header with `default-src 'none'` as baseline. Configure via the `csp` option:

```typescript
csp: {
  scriptSources: ["https://cdn.example.com"],
  connectSources: ["wss://your-domain.com"],
  frameAncestors: ["https://web.telegram.org"],
  allowInline: false,
}
```

## Layer 3: Platform Adapters

### TelegramPlatformAdapter (complete)

Runs client-side inside the Telegram WebView. On `initialize()`:
- Reads `window.Telegram.WebApp`
- Maps `themeParams` entries to CSS custom properties (e.g. `--tg-theme-bg-color`)
- Returns `HostContext` with `theme`, `backgroundColor`, `textColor`, `viewport`, `safeArea`
- Registers Telegram lifecycle callbacks (activate/deactivate)

### LineAdapter (partial)

Extends `BasePostMessageAdapter`. Uses `liff.getAccessToken()` to retrieve the LINE OAuth token. No built-in server-side auth handler — implement your own `BridgeAuthHandler`.

The `LiffSdk` interface expected by `createLineRuntime()`:

```typescript
interface LiffSdk {
  init(config: { liffId: string }): Promise<void>;
  isLoggedIn(): boolean;
  getAccessToken(): string | null;
  sendMessages(messages: object[]): Promise<void>;
  closeWindow(): void;
}
```

## Backend: JsonRpcMcpBackend

`JsonRpcMcpBackend` is the built-in MCP client. It connects to your MCP Server and exposes a single method:

```typescript
const backend = new JsonRpcMcpBackend(config);
const response = await backend.handleMessage(jsonRpcRequest);
```

Pass it to `startResourceServer({ backend })` or call it manually from `onMessage`.

## Resource URI utilities

```typescript
import { parseResourceUri, resolveToHttp } from "@casys/mcp-bridge";

const parsed = parseResourceUri("ui://my-app/dashboard/index.html");
// { server: "my-app", path: "dashboard/index.html" }

const url = resolveToHttp(parsed, "https://bridge.example.com");
// "https://bridge.example.com/app/my-app/dashboard/index.html"
```

<Aside type="note">
  `resolveToHttp` is used internally by bridge.js to convert `ui://` URIs in link-open requests into absolute HTTP URLs before calling `platform.openLink()`.
</Aside>
