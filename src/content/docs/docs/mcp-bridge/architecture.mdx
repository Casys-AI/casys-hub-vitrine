## HTTP surface

`startResourceServer()` exposes these built-in routes:

- `GET /health`
- `POST /session`
- `GET /bridge.js`
- `GET/WS /bridge?session=<id>`
- `GET /app/<server>/<path>`
- `GET /ui?uri=ui://...` when `uiPath` is enabled and `backend.readResource()` exists

Anything not matched by those routes can be handled in `onHttpRequest()`. If that hook returns `{ html, pendingNotifications }`, the server still creates a session, injects the bridge script, and buffers notifications for the new page.

## Browser runtime

The injected runtime in `src/client/bridge.js` is a self-executing script, not just a thin loader.

It:

- reads `platform`, `session`, `auth`, and `debug` from its own script URL
- patches `window.parent.postMessage` in iframe mode and `window.postMessage` in standalone mode
- handles `ui/initialize` locally with host info `{ name: "@casys/mcp-bridge", version: "0.2.0" }`
- handles `ui/open-link` locally through the platform runtime
- tracks pending request IDs with a 30 second timeout
- buffers early incoming notifications until the page has completed `ui/initialize`

## Session and auth flow

1. A session is created before the WebSocket exists.
2. The page loads with `session=<id>` in the injected script URL.
3. The runtime opens `/bridge?session=<id>`.
4. If auth is required, the first WebSocket message must be an auth payload.
5. Only after auth succeeds does the runtime emit `mcp-bridge-ready`.

For Telegram, auth is enforced server-side on the first WebSocket message, not during the initial HTTP request.

## Routing order on the server

For normal JSON-RPC traffic, the server does this:

1. `onMessage(session, message)` if you provided it
2. `backend.handleMessage(session, message)` if you configured a backend
3. JSON-RPC `-32601` for any unhandled request with an `id`

That ordering lets you intercept or short-circuit specific methods without replacing the backend completely.

## Delivery details worth knowing

- Pending notifications stored on the server are flushed as soon as the WebSocket opens.
- The browser runtime will retry the WebSocket connection up to 5 times with exponential backoff capped at 10 seconds.
- Outgoing requests are not queued while disconnected or unauthenticated. They are dropped with a console warning.

<Aside type="caution">
  Root-path assumptions are hard-coded in the injected runtime. `bridge.js` is injected as `/bridge.js?...`, and the WebSocket target is always `/bridge?session=...` at the same origin root.
</Aside>
