<Aside type="caution">
  LINE support is partial in `@casys/mcp-bridge` 0.2.0. The package contains a low-level LINE adapter and a `platform=line` browser runtime branch, but it does not ship a LINE-specific `PlatformAdapter`, LIFF initialization flow, or built-in server-side token validation.
</Aside>

## What exists today

- `LineAdapter` extends the shared `BasePostMessageAdapter`.
- The injected `bridge.js` runtime recognizes `platform=line`.
- That runtime uses a generic mobile host context based on browser APIs.
- If `window.liff.getAccessToken()` exists and returns a value, the runtime can send it as an auth payload over WebSocket.

## What does not exist yet

- no exported `LinePlatformAdapter`
- no built-in LIFF `init()` call
- no LIFF-specific theming or lifecycle mapping
- no built-in LINE access-token validation on the server

## Low-level adapter example

```ts
import { LineAdapter } from "@casys/mcp-bridge";

const adapter = new LineAdapter();

await adapter.init({
  resourceBaseUrl: "https://your-domain.com",
});
```

`LineAdapter` gives you the same low-level shape as the base adapter:

- `init(config)`
- `sendToHost(message)`
- `onMessageFromHost(handler)`
- `destroy()`

## Using the Resource Server with `platform: "line"`

You can still serve HTML and run the browser bridge with:

```ts
import { startResourceServer } from "@casys/mcp-bridge";

const server = startResourceServer({
  assetDirectories: {
    "my-app": "./ui-dist",
  },
  platform: "line",
});
```

If you require auth for LINE, you must supply your own `auth` handler on the server. The package does not provide a LINE validator.

## Practical expectation

Treat the current LINE path as a transport scaffold:

- good enough to experiment with generic webview behavior
- not equivalent to the Telegram path
- not a drop-in LIFF integration
