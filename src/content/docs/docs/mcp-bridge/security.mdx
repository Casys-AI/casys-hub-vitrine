## Telegram authentication (HMAC-SHA256)

Every `POST /session` from a Telegram Mini App is validated before a session is created.

**Algorithm:**

```
1. Parse initData as URL query string -> key-value pairs
2. Remove the "hash" field, keep the rest
3. Sort remaining pairs alphabetically by key
4. Build data_check_string = "key=value\nkey=value\n..." (sorted, newline-separated)
5. secret_key = HMAC-SHA256("WebAppData", botToken)      // note: key is "WebAppData", not the token
6. expected = HMAC-SHA256(data_check_string, secret_key)
7. Compare expected to received hash with timing-safe byte comparison
8. Check auth_date: reject if older than maxAgeSeconds (default 86400 = 24h)
```

All crypto is via `crypto.subtle` (Web Crypto API). No third-party crypto library.

```typescript
import { validateTelegramInitData } from "@casys/mcp-bridge";

const result = await validateTelegramInitData(initDataString, botToken, {
  maxAgeSeconds: 3600, // tighten to 1 hour if needed
});
```

<Aside type="caution">
  Never expose your Telegram bot token to client-side code. Validation runs exclusively on the Resource Server.
</Aside>

## Content Security Policy

`buildCspHeader()` generates a CSP header starting from `default-src 'none'`. Every source must be explicitly listed.

```typescript
import { buildCspHeader } from "@casys/mcp-bridge";

const header = buildCspHeader({
  scriptSources: ["'self'"],
  connectSources: ["wss://bridge.example.com"],
  frameAncestors: ["https://web.telegram.org"],
  allowInline: false,
});
// Content-Security-Policy: default-src 'none'; script-src 'self'; connect-src wss://bridge.example.com; frame-ancestors https://web.telegram.org
```

Configure in `startResourceServer`:

```typescript
startResourceServer({
  csp: {
    scriptSources: ["'self'"],
    connectSources: ["wss://your-domain.com"],
    frameAncestors: ["https://web.telegram.org"],
    allowInline: false,
  },
});
```

## Path traversal protection

Asset serving at `GET /app/<server>/<path>` applies lexical path normalization before resolving to the filesystem:

1. `..` and `.` segments are resolved without any filesystem access
2. The resulting path is checked to confirm it stays within `assetDirectories[server]`
3. Requests that escape the asset directory receive HTTP 403

No symlink resolution is performed.

## Session rate limiting

`SessionStore` enforces a hard limit of **10,000 concurrent sessions**. Once reached, `POST /session` returns HTTP 429 until sessions expire or are cleaned up.

Sessions expire after **30 minutes of inactivity**. A cleanup task runs every **60 seconds**.

```typescript
// SessionStore is accessible on the returned server object
const server = startResourceServer({ ... });
const sessionCount = server.sessions.size;
```

## WebSocket authentication

The session must exist (created via `POST /session`) before a WebSocket connection to `WS /bridge?session=<id>` is accepted. If the session does not exist, the WebSocket is rejected with close code 4401.

When a platform auth handler is configured, the client must send auth as the **first WebSocket message** (before any JSON-RPC traffic). If the auth message is not received within the timeout, the connection is dropped.

## Allowed origins

Restrict which origins can connect to the WebSocket:

```typescript
startResourceServer({
  options: {
    allowedOrigins: ["https://your-domain.com"],
  },
});
```

Requests with an `Origin` header not in the allowlist are rejected with HTTP 403.

## Summary

| Threat | Mitigation |
|--------|------------|
| Forged Telegram initData | HMAC-SHA256 validation, timing-safe comparison, 24h max age |
| XSS via served HTML | CSP `default-src 'none'`, explicit allowlists |
| Path traversal on asset serving | Lexical normalization, boundary check |
| Session flooding | Hard cap of 10,000 sessions, HTTP 429 |
| Stale sessions | 30-minute TTL, 60-second cleanup interval |
| Unauthorized WebSocket | Session required before WS connect; auth-on-connect if platform auth configured |
| Cross-origin WebSocket | `allowedOrigins` checked on upgrade request |
