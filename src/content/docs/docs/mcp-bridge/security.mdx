## Sessions

Sessions are created in memory with 128 bits of randomness from `crypto.getRandomValues()`. A session exists before the WebSocket handshake:

- when the server serves HTML from `/app/...`
- when the server serves proxied HTML from `onHttpRequest()` or `/ui?uri=...`
- when a client explicitly calls `POST /session`

The server stores:

- `id`
- `platform`
- timestamps
- `authenticated`
- optional `principalId`, `userId`, `username`, and `authContext`
- optional buffered notifications

The default session TTL is 30 minutes, and expired sessions are cleaned every 60 seconds.

<Aside type="note">
  Session TTL is not configurable through `startResourceServer()` in the current source. The server always constructs `new SessionStore()` with the default max age.
</Aside>

## Auth flow

Auth is optional and generic. If an auth handler is configured, the first WebSocket message must be an auth payload.

For Telegram, `telegramBotToken` installs the built-in auth handler. An unauthenticated non-auth message causes:

- an `auth_error` frame
- WebSocket close code `4003`

Failed auth closes the socket with `4001`. Expired sessions are closed with `4002`.

## Telegram `initData` validation

The built-in Telegram validator:

- requires `hash`
- requires `auth_date`
- rejects invalid hex
- verifies the HMAC-SHA256 signature with Web Crypto
- rejects `auth_date` older than 24 hours
- rejects `auth_date` more than 60 seconds in the future

If the `user` field is present, the validator also extracts `userId`, `username`, `firstName`, and `lastName`.

## Content Security Policy

HTML responses get a CSP built from `buildCspHeader()`. The default policy is:

```text
default-src 'none';
script-src 'self' 'unsafe-inline';
style-src 'self' 'unsafe-inline';
img-src 'self' data:;
font-src 'self';
connect-src 'self';
frame-ancestors 'self';
base-uri 'self'
```

Extra sources come from:

- `config.csp` for all served HTML
- `UiResourceResponse.csp` only for backend-proxied resources

Local files under `/app/...` do not have per-resource metadata-driven CSP in the current implementation.

## Filesystem protections

Path traversal protection for `/app/...` is lexical:

- `.` and `..` are normalized in string space
- the resolved path must still start with the configured base directory

This protects against simple traversal attempts, but it does not resolve symlinks.

## Current trust boundaries

The package has important security limits that the docs should not hide:

- `allowedOrigins` only affects HTTP CORS headers. The WebSocket endpoint does not perform explicit `Origin` validation.
- The shared `BasePostMessageAdapter` constrains outbound `targetOrigin` from `resourceBaseUrl`, but it does not validate inbound `message` events by `event.origin`.
- Session state is in-memory only. Restarting the server clears sessions and buffered notifications.
- There is no built-in LINE token validation.
