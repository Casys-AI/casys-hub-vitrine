## Two browser-side layers

`@casys/mcp-bridge` exposes two related browser-side pieces:

1. The injected `bridge.js` runtime that `startResourceServer()` serves at `/bridge.js`
2. The exported `BridgeClient` class in `src/core/bridge-client.ts`

They solve the same problem, but they are not identical implementations.

## Injected `bridge.js`

The Resource Server injects a script URL shaped like:

```text
/bridge.js?platform=<platform>&session=<sessionId>[&auth=1][&debug=1]
```

That runtime:

- reads its own query parameters from `document.currentScript`
- patches `window.parent.postMessage` in iframe mode, or `window.postMessage` in standalone mode
- handles `ui/initialize` locally
- handles `ui/open-link` locally
- forwards everything else over WebSocket
- emits `mcp-bridge-ready`
- emits `mcp-bridge-auth-error`

### Runtime behavior

- `ui/initialize` returns protocol version `2025-11-25`
- host info is `{ name: "@casys/mcp-bridge", version: "0.2.0" }`
- pending request IDs time out after 30 seconds
- early incoming notifications are buffered until `ui/initialize` completes
- WebSocket reconnect is attempted up to 5 times with exponential backoff up to 10 seconds
- outgoing messages are dropped if the socket is closed or auth has not completed yet

## Exported `BridgeClient`

Use the exported class if you are composing your own platform runtime instead of relying on the injected script.

```ts
import {
  BridgeClient,
  TelegramPlatformAdapter,
  WebSocketTransport,
} from "@casys/mcp-bridge";

const client = new BridgeClient({
  serverUrl: "https://bridge.example.com",
  sessionId: "session-id",
  platform: new TelegramPlatformAdapter(),
  transport: new WebSocketTransport(),
});

await client.start();
```

## `BridgeClientOptions`

| Field | Type | Notes |
| --- | --- | --- |
| `serverUrl` | `string` | Base server URL. `BridgeClient` appends `/bridge?session=<id>`. |
| `platform` | `PlatformAdapter` | Required. Supplies host context, lifecycle events, and `openLink()`. |
| `transport` | `BridgeTransport` | Required in code. The package exports `WebSocketTransport`. |
| `sessionId` | `string` | Required. Passed to the WebSocket URL. |
| `bridgeInfo` | `{ name: string; version: string }` | Optional override for reported host info. |
| `requestTimeoutMs` | `number` | Optional. Defaults to `30000`. |

## `BridgeClient` behavior

The class:

- initializes the platform adapter
- connects the transport
- intercepts `window.parent.postMessage`
- handles `ui/initialize` locally
- handles `ui/open-link` locally
- forwards the rest through the transport
- translates `theme-changed` and `viewport-changed` lifecycle events into `ui/notifications/host-context-changed`
- dispatches `ui/resource-teardown` on a `teardown` lifecycle event

## Important differences from the injected runtime

- The exported `BridgeClient` does not implement the injected runtime's reconnect loop.
- The exported `BridgeClient` does not implement the injected runtime's auth message protocol.
- Only `WebSocketTransport` is implemented in the package today.
