## API

```typescript
import { buildCompositeUi } from "@casys/mcp-compose";
```

```typescript
function buildCompositeUi(
  resources: CollectedUiResource[],
  orchestration?: UiOrchestration,
): CompositeUiDescriptor
```

## What it returns

`buildCompositeUi()` produces:

```typescript
interface CompositeUiDescriptor {
  type: "composite";
  resourceUri: string;
  layout: UiLayout;
  children: CollectedUiResource[];
  sync: ResolvedSyncRule[];
  sharedContext?: Record<string, unknown>;
  areaMap?: Record<string, string>;
}
```

Important implementation details:

- `type` is always `"composite"`.
- `resourceUri` is always `ui://mcp-compose/workflow/<uuid>`.
- `children` are the original collected resources in order.
- `layout` defaults to `"stack"` when orchestration is omitted.
- `sync` contains resolved slot-based rules, not tool-name rules.

## Basic example

```typescript
const descriptor = buildCompositeUi(
  [
    { source: "postgres:query", resourceUri: "ui://pg/1", slot: 0 },
    { source: "viz:render", resourceUri: "ui://viz/2", slot: 1 },
  ],
  {
    layout: "split",
    sync: [
      { from: "postgres:query", event: "filter", to: "viz:render", action: "update" },
    ],
    sharedContext: ["workflowId"],
  },
);

// descriptor.sync[0] === { from: 0, event: "filter", to: 1, action: "update" }
```

## `UiOrchestration`

```typescript
interface UiOrchestration {
  layout: UiLayout;
  sync?: UiSyncRule[];
  sharedContext?: string[];
}
```

### Layout presets

The preset string layouts are:

- `"split"`: side-by-side flex panels
- `"tabs"`: tab bar with one visible iframe at a time
- `"grid"`: auto-fit CSS grid
- `"stack"`: vertical stack

### Areas layout

`UiLayout` can also be an object:

```typescript
const layout = {
  areas: [
    ["filter", "list", "detail"],
    ["filter", "chart", "chart"],
  ],
  columns: [1, 2, 2],
  rows: [3, 1],
  gap: "normal" as const,
};
```

The renderer treats repeated area names as CSS grid spans. `gap` is a token, not a number: `"none"`, `"compact"`, `"normal"`, or `"spacious"`.

## Shared context

When you pass `sharedContext`, the composer scans resource `context` objects in order and takes the first value it finds for each requested key:

```typescript
const descriptor = buildCompositeUi(
  [
    {
      source: "postgres:query",
      resourceUri: "ui://pg/1",
      slot: 0,
      context: { workflowId: "wf-123", query: "SELECT *" },
    },
    {
      source: "viz:render",
      resourceUri: "ui://viz/2",
      slot: 1,
      context: { workflowId: "wf-999", userId: "user-456" },
    },
  ],
  {
    layout: "split",
    sharedContext: ["workflowId", "userId"],
  },
);

// descriptor.sharedContext === {
//   workflowId: "wf-123",
//   userId: "user-456",
// }
```

This is first-match-wins, not last-write-wins.

## Sync resolution

`buildCompositeUi()` calls `resolveSyncRules()` internally. That means:

- valid tool-name rules become slot-based rules
- `to: "*"` stays `"*"`
- unknown `from` or `to` references are excluded from the final descriptor

If you want to surface bad sync rules to callers, run [`validateSyncRules`](./sync-rules/) before composing.

## `areaMap`

The descriptor type includes `areaMap`, but `buildCompositeUi()` does not populate it. In this repo, `areaMap` is added later by the runtime layer when a dashboard template assigns source IDs for areas layouts.
