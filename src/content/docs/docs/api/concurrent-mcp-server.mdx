The main entry point for the framework. `McpApp` handles tool registration, middleware composition, transport selection, and lifecycle management.

<Aside type="note">
  Deno-first. Node.js 20+ is supported via the JSR npm compatibility layer.
</Aside>

## Constructor

```typescript
import { McpApp } from "@casys/mcp-server";

const server = new McpApp(options: McpAppOptions);
```

### McpAppOptions

`ConcurrentServerOptions` is a deprecated alias for `McpAppOptions` and will be removed in v1.0. Use `McpAppOptions` in new code.

```typescript
interface McpAppOptions {
  // Required
  name: string;                          // Server name (shown to clients)
  version: string;                       // Semver version string

  // Concurrency
  maxConcurrent?: number;                // Default: 10
  backpressureStrategy?: "sleep" | "queue" | "reject";  // Default: "sleep"
  backpressureSleepMs?: number;          // Sleep interval for "sleep" strategy (ms)

  // Rate limiting
  rateLimit?: RateLimitOptions;

  // Schema validation
  validateSchema?: boolean;              // Default: false — enables ajv 8

  // Sampling
  enableSampling?: boolean;              // Enable bidirectional LLM delegation
  samplingClient?: SamplingClient;       // External sampling client (alternative to bridge)

  // Resources
  expectResources?: boolean;             // Enable dynamic resource support

  // Auth
  auth?: AuthOptions;                    // OAuth2/Bearer config

  // MCP Apps
  resourceCsp?: CspOptions;             // CSP options for HTML resource injection

  // Customization
  instructions?: string;                // Instructions sent to clients at initialization
  toolErrorMapper?: (error: unknown, toolName: string) => string | null;  // Return message to surface as isError:true, or null to rethrow
  logger?: (msg: string) => void;       // Default: console.error
}
```

### RateLimitOptions

```typescript
interface RateLimitOptions {
  maxRequests: number;    // Max requests per window
  windowMs: number;       // Window size in milliseconds
  keyExtractor?: (context: RateLimitContext) => string;  // Default: "default" (global)
  onLimitExceeded?: "reject" | "wait";  // Default: "wait"
}

interface RateLimitContext {
  toolName: string;
  args: Record<string, unknown>;
}
```

### AuthOptions

```typescript
interface AuthOptions {
  provider: AuthProvider;              // Use presets (createGoogleAuthProvider, etc.) or JwtAuthProvider
}
```

Pass `resource`, `authorizationServers`, and `scopesSupported` to the preset factory instead — they live on the provider, not on `AuthOptions`.

## Registration methods

All registration must happen **before** calling `start()` or `startHttp()`. Exception: `registerToolLive()` is designed for post-start registration. Setting `expectResources: true` also enables dynamic resource registration after start.

### registerTool

```typescript
server.registerTool(tool: MCPTool, handler: ToolHandler): void;
```

Register a single tool with its handler:

```typescript
server.registerTool(
  {
    name: "greet",
    description: "Greet someone",
    inputSchema: {
      type: "object",
      properties: { name: { type: "string" } },
      required: ["name"],
    },
    requiredScopes: ["read"],   // Optional — requires this scope if auth is configured
    outputSchema: {             // Optional — for structured tool results (SDK 1.27)
      type: "object",
      properties: { message: { type: "string" } },
    },
    annotations: {              // Optional — hints for clients
      readOnlyHint: true,
    },
  },
  ({ name }) => `Hello, ${name}!`,
);
```

### registerTools

```typescript
server.registerTools(tools: MCPTool[], handlers: Map<string, ToolHandler>): void;
```

Bulk-register tools. Map keys must match tool names exactly.

### registerToolLive

```typescript
server.registerToolLive(tool: MCPTool, handler: ToolHandler): void;
```

Register a tool after the server has started. Clients will see the new tool on their next `tools/list` request.

### unregisterTool

```typescript
server.unregisterTool(toolName: string): void;
```

Remove a registered tool. Clients will no longer see it on subsequent `tools/list` requests.

### registerAppOnlyTool

```typescript
server.registerAppOnlyTool(tool: MCPTool, handler: ToolHandler): void;
```

Register a tool visible to UI clients only — it does not appear in the LLM's tool list.

### registerResource

```typescript
server.registerResource(resource: MCPResource, handler: ResourceHandler): void;
```

Register a single resource. See [MCP Apps](/docs/guides/mcp-apps) for `text/html;profile=mcp-app` resources.

### registerResources

```typescript
server.registerResources(
  resources: MCPResource[],
  handlers: Map<string, ResourceHandler>,
): void;
```

Bulk-register resources.

### registerViewers

```typescript
server.registerViewers(config: RegisterViewersConfig): void;
```

Bulk-register file-based viewer resources. Each entry maps a URI prefix to a module configuration:

```typescript
server.registerViewers({
  prefix: "ui://my-server/",
  moduleUrl: import.meta.url,
  exists: async (path) => /* check if path exists */,
  readFile: async (path) => /* return file contents as string */,
});
```

The `prefix` is the base URI for all registered resources. `moduleUrl` is used to resolve relative paths. `exists` and `readFile` are called by the framework to serve viewer files.

### use

```typescript
server.use(middleware: Middleware): this;
```

Add a custom middleware to the pipeline. Returns `this` for chaining:

```typescript
server
  .use(logging)
  .use(metrics)
  .use(caching);
```

<Aside type="caution">
  Throws if called after `start()` or `startHttp()`. The pipeline is compiled at startup.
</Aside>

See the [Middleware Pipeline guide](/docs/guides/middleware) for patterns and recipes.

## Transport methods

### start

```typescript
await server.start(): Promise<void>;
```

Start with STDIO transport. Use this for local tools (Claude Desktop, Cursor, etc.). Blocks until the client disconnects.

### startHttp

```typescript
async startHttp(options: HttpServerOptions): Promise<HttpServerInstance>
```

Start with Streamable HTTP transport. Includes SSE streaming, session management, and RFC 9728 discovery. Returns a `Promise<HttpServerInstance>` — always `await` it.

```typescript
interface HttpServerOptions {
  port: number;
  hostname?: string;              // Default: "0.0.0.0"
  cors?: boolean;                 // Default: true
  corsOrigins?: "*" | string[];   // Default: "*" — use allowlist in production
  maxBodyBytes?: number | null;   // Default: 1_000_000 (1 MB). null = no limit
  requireAuth?: boolean;          // Default: false — throws at startup if true without auth
  customRoutes?: RouteConfig[];   // Additional HTTP routes
  ipRateLimit?: HttpRateLimitOptions;
  onListen?: (info: { hostname: string; port: number }) => void;  // Called when server is ready
  embedded?: boolean;             // Skip port binding — use getFetchHandler() instead
  embeddedHandlerCallback?: (handler: FetchHandler) => void;  // Receives handler in embedded mode (set internally by getFetchHandler)
}

interface HttpRateLimitOptions {
  maxRequests: number;
  windowMs: number;
  keyExtractor?: (context: HttpRateLimitContext) => string;  // Custom key extraction (default: IP)
  onLimitExceeded?: "reject" | "wait";  // "reject" returns 429 immediately; "wait" queues the request until a slot opens
}

interface HttpRateLimitContext {
  ip: string;
  method: string;
  path: string;
  headers: Headers;
  sessionId?: string;
}
```

HTTP routes served automatically:

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/mcp` | JSON-RPC tool calls |
| `GET` | `/mcp` | SSE stream for notifications |
| `GET` | `/health` | Health check |
| `GET` | `/metrics` | Prometheus metrics |
| `GET` | `/.well-known/oauth-protected-resource` | RFC 9728 discovery (if auth configured) |

Sessions: 32-character hex IDs, 30-minute TTL, max 10,000 active sessions, per-IP rate limit of 10 `initialize` calls per 60 seconds.

**Example with security options:**

```typescript
await server.startHttp({
  port: 3000,
  requireAuth: true,
  corsOrigins: ["https://app.example.com"],
  maxBodyBytes: 1_048_576,
  ipRateLimit: { maxRequests: 60, windowMs: 60_000 },
  onListen: ({ hostname, port }) => console.log(`Ready on ${hostname}:${port}`),
});
```

### getFetchHandler

```typescript
await server.getFetchHandler(
  options?: Omit<HttpServerOptions, "port" | "hostname" | "onListen">
): Promise<FetchHandler>;
```

Build the full HTTP middleware stack and return its Web Standard fetch handler **without binding a port**. Use this to embed the MCP HTTP layer inside an existing web framework (Fresh, Hono, Express) instead of giving up port ownership to `startHttp`.

The returned handler accepts a `Request` and returns a `Response`. It exposes the same routes as `startHttp`: `POST /mcp`, `GET /mcp` (SSE), `GET /health`, `GET /metrics`, and `GET /.well-known/oauth-protected-resource`. Auth, rate limiting, session cleanup, and OTel hooks all work identically.

```typescript
import { Hono } from "hono";

const app = new Hono();
const mcpApp = new McpApp({ name: "my-app", version: "1.0.0" });
// ... register tools ...
const handler = await mcpApp.getFetchHandler();
app.all("/mcp/*", (c) => handler(c.req.raw));
```

Multi-tenant SaaS pattern: instantiate one `McpApp` per tenant, call `getFetchHandler()` once per server, then dispatch each inbound request to the right handler from your router.

### stop

```typescript
await server.stop(): Promise<void>;
```

Graceful shutdown. Waits for in-flight requests to complete before closing.

## Notification methods

### sendNotification

```typescript
server.sendNotification(method: string, params?: Record<string, unknown>): void;
```

Send a notification to the current session (STDIO transport).

### broadcastNotification

```typescript
server.broadcastNotification(
  method: string,
  params?: Record<string, unknown>,
): void;
```

Push a notification to **all** connected SSE clients (HTTP transport).

### sendToSession

```typescript
server.sendToSession(sessionId: string, message: Record<string, unknown>): void;
```

Push a message to a specific SSE session by ID (HTTP transport).

### getPrometheusMetrics

```typescript

```

Returns metrics in Prometheus text format. Same data served at `GET /metrics`. All metric names use the `mcp_server_` prefix:

```
# HELP mcp_server_tool_calls_total Total number of tool calls
# TYPE mcp_server_tool_calls_total counter
mcp_server_tool_calls_total{tool="greet",status="success"} 42

# HELP mcp_server_tool_call_duration_ms Tool call duration in milliseconds
# TYPE mcp_server_tool_call_duration_ms histogram
mcp_server_tool_call_duration_ms_bucket{tool="greet",le="100"} 40
...

# HELP mcp_server_active_requests Currently active requests
# TYPE mcp_server_active_requests gauge
mcp_server_active_requests 3

# HELP mcp_server_active_sessions Currently active SSE sessions
# TYPE mcp_server_active_sessions gauge
mcp_server_active_sessions 7
```

Counters available: `mcp_server_tool_calls_total`, `mcp_server_auth_success_total`, `mcp_server_auth_failed_total`.

### getRateLimiter

### getResourceInfo

```typescript
server.getResourceInfo(uri: string): MCPResource | undefined;
```

Returns the `MCPResource` definition (uri, name, description, mimeType) for a registered resource, or `undefined` if not found. Useful for testing and debugging.

### readResourceContent

```typescript
server.readResourceContent(uri: string): Promise<ResourceContent | null>;
```

Invoke a registered resource handler directly and return its content, without going through the MCP protocol. Returns `null` if the resource is not registered. Useful for server-side rendering or integration tests.

```typescript
const content = await server.readResourceContent("ui://my-server/viewer");
// content.text contains the HTML
```

### isStarted

```typescript
server.isStarted(): boolean;
```

Returns `true` after `start()` or `startHttp()` has been called.

## Multi-tenant middleware

For SaaS deployments where each authenticated user belongs to a tenant, use `createMultiTenantMiddleware` to inject a `tenantId` into the middleware context. It must run **after** the auth middleware (which populates `ctx.authInfo`) and before scope enforcement or tool handlers.

```typescript
import {
  createMultiTenantMiddleware,
  type TenantResolver,
  type TenantResolution,
} from "@casys/mcp-server";

class SubdomainTenantResolver implements TenantResolver {
  async resolve(ctx): Promise<TenantResolution> {
    const host = ctx.request!.headers.get("host") ?? "";
    const subdomain = host.split(".")[0];
    const claim = (ctx.authInfo as any).claims?.["urn:my-app:tenant_id"];
    if (typeof claim !== "string" || claim !== subdomain) {
      return { ok: false, reason: `claim/subdomain mismatch` };
    }
    return { ok: true, tenantId: subdomain };
  }
}

server.use(createMultiTenantMiddleware(new SubdomainTenantResolver(), {
  onRejection: (ctx, reason) => auditLog.warn("tenant_reject", { reason }),
}));
```

After this middleware runs, `ctx.authInfo.tenantId` is set and frozen. All downstream middlewares and tool handlers can read it to scope data access.

### Types

```typescript
type TenantResolution =
  | { ok: true; tenantId: string }
  | { ok: false; reason: string };

interface TenantResolver {
  resolve(ctx: MiddlewareContext): Promise<TenantResolution>;
}

interface MultiTenantMiddlewareOptions {
  /**
   * Called on rejection before the 401 is sent.
   * Use for audit logging. The reason is never exposed to clients.
   * If this hook throws, the error is logged to stderr but the 401 still fires.
   */
  onRejection?: (ctx: MiddlewareContext, reason: string) => void | Promise<void>;
}
```

Pipeline behaviour:
- **STDIO transport** (no `ctx.request`): passes through unchanged — STDIO has no tenant concept.
- **HTTP without `ctx.authInfo`**: throws a configuration error. The auth middleware must precede this one.
- **HTTP with rejected resolution**: calls `onRejection`, then throws `AuthError("invalid_token")` — a generic 401 with no tenant details leaked to the client.

## Sampling bridge

When `enableSampling: true` is set, the server can delegate LLM calls back through the client connection. Provide a `samplingClient` — either instantiate `SamplingBridge` directly or supply any object implementing the `SamplingClient` interface:

```typescript
import { McpApp, SamplingBridge } from "@casys/mcp-server";
import type { SamplingClient } from "@casys/mcp-server";

// Option A: use SamplingBridge with your own SamplingClient implementation
const myClient: SamplingClient = {
  async createMessage(params) {
    // delegate to your LLM provider
  },
};
const samplingClient = new SamplingBridge(myClient);

const server = new McpApp({
  name: "my-server",
  version: "1.0.0",
  enableSampling: true,
  samplingClient,
});

server.registerTool(
  { name: "summarize", description: "Summarize text", inputSchema: { type: "object", properties: { text: { type: "string" } } } },
  async (args) => {
    const result = await samplingClient.requestSampling({
      messages: [{ role: "user", content: args.text as string }],
      maxTokens: 500,
      maxIterations: 3,
    });
    return result.content;
  },
);
```

Note: `ToolHandler` only receives `args` — there is no `{ bridge }` second argument.

## OpenTelemetry

Set `OTEL_DENO=true` to enable OpenTelemetry tracing. Spans are emitted for tool calls and auth events.
