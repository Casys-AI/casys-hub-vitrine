The building blocks of `@casys/mcp-server` are useful on their own. Rate limiting, concurrency control, and schema validation ŌĆö all usable in any Deno or Node.js project, no MCP required.

## RateLimiter

Sliding-window rate limiter with per-key tracking. No external dependencies.

```typescript
import { RateLimiter } from "@casys/mcp-server";

const limiter = new RateLimiter({
  maxRequests: 10,   // 10 requests
  windowMs: 1000,    // per 1 second
});
```

### checkLimit

```typescript
limiter.checkLimit(key: string): boolean;
```

Returns `true` if within limits, `false` if rate-limited. Non-blocking ŌĆö never waits.

```typescript
if (limiter.checkLimit("client-123")) {
  // Process request
} else {
  // Rate limited
}
```

### waitForSlot

```typescript
await limiter.waitForSlot(key: string): Promise<void>;
```

Blocks until a slot opens. Maximum wait time equals `windowMs`.

```typescript
await limiter.waitForSlot("client-123");
// Guaranteed to have a slot when this resolves
```

### getRemainingRequests

```typescript
limiter.getRemainingRequests(key: string): number;
```

Returns how many requests the given key can still make in the current window. Returns `maxRequests` for keys with no history.

```typescript
const remaining = limiter.getRemainingRequests("client-123");
// e.g., 7 if 3 of 10 requests have been used
```

### getMetrics

```typescript
limiter.getMetrics(): { keys: number; totalRequests: number };
```

## RequestQueue

Concurrency control with configurable backpressure. Protects downstream services from overload.

```typescript
import { RequestQueue } from "@casys/mcp-server";

const queue = new RequestQueue({
  maxConcurrent: 5,
  strategy: "queue",     // "sleep" | "queue" | "reject"
  sleepMs: 10,           // For "sleep" strategy only ŌĆö default: 10
});
```

### Strategies

| Strategy | Behavior | Best for |
|----------|----------|----------|
| `sleep` | Spin-wait with `sleepMs` delay until a slot opens | Low-latency, CPU-tolerant |
| `queue` | FIFO queue ŌĆö requests wait in order | Fair ordering, predictable latency |
| `reject` | Immediately reject if at capacity | Fail-fast, circuit-breaker style |

### acquire / release

```typescript
await queue.acquire(): Promise<void>;
queue.release(): void;
```

Always use try/finally to prevent slot leaks:

```typescript
await queue.acquire();
try {
  await processRequest();  // 1 of maxConcurrent slots occupied
} finally {
  queue.release();
}
```

### getMetrics

```typescript
queue.getMetrics(): { inFlight: number; queued: number };
```

## SchemaValidator

JSON Schema validation powered by ajv 8 with `allErrors` and `useDefaults`. Schemas are compiled once at `addSchema()` time ŌĆö validation calls are fast.

```typescript
import { SchemaValidator } from "@casys/mcp-server";

const validator = new SchemaValidator();
```

### addSchema

```typescript
validator.addSchema(name: string, schema: object): void;
```

Compiles and caches the schema. Call once at startup:

```typescript
validator.addSchema("my_tool", {
  type: "object",
  properties: {
    count: { type: "number", minimum: 0 },
    label: { type: "string", maxLength: 100 },
  },
  required: ["count"],
});
```

### validate

```typescript
validator.validate(name: string, data: unknown): ValidationResult;
```

```typescript
interface ValidationResult {
  valid: boolean;
  errors: string[];  // Human-readable error messages (empty if valid)
}
```

```typescript
validator.validate("my_tool", { count: -1 });
// { valid: false, errors: ["count must be >= 0"] }

validator.validate("my_tool", { count: 5 });
// { valid: true, errors: [] }
```

### validateOrThrow

```typescript
validator.validateOrThrow(name: string, data: unknown): void;
```

Throws a `ValidationError` if the data is invalid. Useful when you want to fail fast rather than handle a result object:

```typescript
try {
  validator.validateOrThrow("my_tool", args);
  // args are valid ŌĆö proceed
} catch (error) {
  // error.message contains all validation errors
}
```

<Aside type="tip">
  Inside `ConcurrentMCPServer`, these components are wired up automatically based on your options. You only need standalone usage when building custom infrastructure outside the framework.
</Aside>
