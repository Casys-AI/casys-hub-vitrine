`@casys/mcp-bridge` transforme une interface MCP App en une page HTTP qui peut s'exécuter dans une webview de plateforme de messagerie. Dans le package actuel, cela signifie :

1. Un Resource Server sert du HTML depuis des répertoires d'assets locaux ou un résolveur `ui://` backend.
2. Le serveur injecte `/bridge.js?...` dans les réponses HTML.
3. Le runtime injecté déplace les messages JSON-RPC MCP entre la page et le serveur via `/bridge?session=...`.

Si votre interface parle déjà à son hôte avec MCP Apps JSON-RPC via `postMessage`, le bridge conserve ce contrat et gère le transport webview autour de lui.

## Ce qui est implémenté

<CardGrid>
  <Card title="Resource Server" icon="server">
    `startResourceServer()` sert `/app`, `/bridge.js`, `/bridge`, `/health`, `/session`, et optionnellement `/ui`.
  </Card>
  <Card title="Runtime Navigateur" icon="puzzle">
    Le `bridge.js` injecté gère `ui/initialize` et `ui/open-link` localement, puis transfère le reste via WebSocket.
  </Card>
  <Card title="Telegram" icon="rocket">
    Telegram est le seul parcours de plateforme de bout en bout avec une auth serveur intégrée basée sur `initData` de Mini App.
  </Card>
  <Card title="LINE" icon="setting">
    LINE est partiel aujourd'hui : il existe un adaptateur bas niveau et une branche de runtime navigateur, mais pas d'initialisation LIFF intégrée ni de validation de token.
  </Card>
</CardGrid>

## Ce que le package fait réellement

- Le HTML local sous `assetDirectories` est servi sur `/app/<server>/<path>`.
- Les ressources `ui://` résolues par le backend peuvent être exposées sur `/ui?uri=ui://...` quand votre backend implémente `readResource()`.
- Chaque page HTML servie reçoit un nouvel ID de session en mémoire et un script bridge injecté.
- Le script bridge gère `ui/initialize` lui-même et rapporte la version de protocole `2025-11-25`.
- Tout autre trafic JSON-RPC va au serveur via WebSocket, où `onMessage()` s'exécute en premier et `backend.handleMessage()` en second.

## Limites importantes

- Le script injecté est référencé comme `/bridge.js?...`, et le runtime navigateur se connecte toujours à `/bridge?session=...` à la racine de même origine. Montez le bridge à la racine de l'origine ou ajoutez des réécritures au niveau racine.
- Le package n'exporte qu'un `WebSocketTransport`. Il n'y a pas de transport de repli HTTP implémenté.
- L'état de session est uniquement en mémoire.
- LINE n'est pas à parité de fonctionnalités avec Telegram.
