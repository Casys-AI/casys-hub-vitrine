Les briques de base de `@casys/mcp-server` sont utiles seules. Rate limiting, controle de concurrence, et validation de schema — tout utilisable dans n'importe quel projet Deno ou Node.js, sans MCP.

## RateLimiter

Rate limiter a fenetre glissante avec suivi par cle. Aucune dependance externe.

```typescript
import { RateLimiter } from "@casys/mcp-server";

const limiter = new RateLimiter({
  maxRequests: 10,   // 10 requests
  windowMs: 1000,    // per 1 second
});
```

### checkLimit

```typescript
limiter.checkLimit(key: string): boolean;
```

Retourne `true` si dans les limites, `false` si rate-limited. Non-bloquant — n'attend jamais.

```typescript
if (limiter.checkLimit("client-123")) {
  // Process request
} else {
  // Rate limited
}
```

### waitForSlot

```typescript
await limiter.waitForSlot(key: string): Promise<void>;
```

Bloque jusqu'a ce qu'un slot se libere. Le temps d'attente maximum est egal a `windowMs`.

```typescript
await limiter.waitForSlot("client-123");
// Guaranteed to have a slot when this resolves
```

### getRemainingRequests

```typescript
limiter.getRemainingRequests(key: string): number;
```

Retourne le nombre de requetes que la cle donnee peut encore effectuer dans la fenetre courante. Retourne `maxRequests` pour les cles sans historique.

```typescript
const remaining = limiter.getRemainingRequests("client-123");
// e.g., 7 if 3 of 10 requests have been used
```

### getTimeUntilSlot

```typescript
limiter.getTimeUntilSlot(key: string): number;
```

Retourne le nombre de millisecondes avant l'ouverture du prochain slot pour la cle donnee. Retourne `0` si un slot est deja disponible.

### clear

```typescript
limiter.clear(key: string): void;
```

Efface l'historique des requetes pour une cle specifique, liberant immediatement ses slots.

### clearAll

```typescript
limiter.clearAll(): void;
```

Efface toutes les cles suivies et leur historique de requetes.

### getMetrics

```typescript
limiter.getMetrics(): { keys: number; totalRequests: number };
```

## RequestQueue

Controle de concurrence avec backpressure configurable. Protege les services en aval de la surcharge.

```typescript
import { RequestQueue } from "@casys/mcp-server";

const queue = new RequestQueue({
  maxConcurrent: 5,
  strategy: "queue",     // "sleep" | "queue" | "reject"
  sleepMs: 10,           // For "sleep" strategy only — default: 10
});
```

### Strategies

| Strategie | Comportement | Ideal pour |
|-----------|-------------|------------|
| `sleep` | Attente active avec delai `sleepMs` jusqu'a liberation d'un slot | Faible latence, tolerant au CPU |
| `queue` | File FIFO — les requetes attendent dans l'ordre | Ordonnancement equitable, previsible |
| `reject` | Rejet immediat si a pleine capacite | Fail-fast, style circuit-breaker |

### acquire / release

```typescript
await queue.acquire(): Promise<void>;
queue.release(): void;
```

Toujours utiliser try/finally pour eviter les fuites de slots :

```typescript
await queue.acquire();
try {
  await processRequest();  // 1 of maxConcurrent slots occupied
} finally {
  queue.release();
}
```

### getMetrics

```typescript
queue.getMetrics(): { inFlight: number; queued: number };
```

## SchemaValidator

Validation JSON Schema propulsee par ajv 8 avec `allErrors` et `useDefaults`. Les schemas sont compiles une seule fois a l'appel de `addSchema()` — les appels de validation sont rapides.

```typescript
import { SchemaValidator } from "@casys/mcp-server";

const validator = new SchemaValidator();
```

### addSchema

```typescript
validator.addSchema(name: string, schema: object): void;
```

Compile et met en cache le schema. A appeler une fois au demarrage :

```typescript
validator.addSchema("my_tool", {
  type: "object",
  properties: {
    count: { type: "number", minimum: 0 },
    label: { type: "string", maxLength: 100 },
  },
  required: ["count"],
});
```

### validate

```typescript
validator.validate(name: string, data: unknown): ValidationResult;
```

```typescript
interface ValidationResult {
  valid: boolean;
  errors: ValidationError[];  // Structured error objects (empty if valid)
}
```

```typescript
validator.validate("my_tool", { count: -1 });
// { valid: false, errors: ["count must be >= 0"] }

validator.validate("my_tool", { count: 5 });
// { valid: true, errors: [] }
```

### hasSchema

```typescript
validator.hasSchema(name: string): boolean;
```

Retourne `true` si un schema avec le nom donne a ete ajoute.

### removeSchema

```typescript
validator.removeSchema(name: string): void;
```

Supprime un schema precedemment ajoute. Les appels subsequents a `validate()` avec ce nom lanceront une erreur.

### validateOrThrow

```typescript
validator.validateOrThrow(name: string, data: unknown): void;
```

Lance une `ValidationError` si les donnees sont invalides. Utile quand vous voulez echouer rapidement plutot que de gerer un objet resultat :

```typescript
try {
  validator.validateOrThrow("my_tool", args);
  // args are valid — proceed
} catch (error) {
  // error.message contains all validation errors
}
```

<Aside type="tip">
  Dans `McpApp`, ces composants sont cables automatiquement selon vos options. L'utilisation standalone n'est necessaire que pour construire une infrastructure personnalisee en dehors du framework.
</Aside>
