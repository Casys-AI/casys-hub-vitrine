## API

```typescript
import { renderComposite } from "@casys/mcp-compose";
```

L'export racine utilise le renderer host de `@casys/mcp-compose/host`.

```typescript
function renderComposite(descriptor: CompositeUiDescriptor): string
```

## Ce que ca genere

`renderComposite()` retourne un document HTML :

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>mcp-compose</title>
  <style>...</style>
</head>
<body>
  <div class="layout-split" id="container">
    <iframe id="ui-0" src="ui://postgres/table/123" data-slot="0" data-source="postgres:query"></iframe>
    <iframe id="ui-1" src="ui://viz/chart/456" data-slot="1" data-source="viz:chart"></iframe>
  </div>
  <script>...</script>
</body>
</html>
```

Le renderer echappe les labels des onglets et les valeurs d'attributs des iframes avant de les inserer dans le HTML.

## Comportement des layouts

Le renderer public supporte :

- `split` : `display: flex; height: 100vh`
- `tabs` : barre d'onglets avec iframes en position absolue et premier onglet actif
- `grid` : `display: grid` avec `repeat(auto-fit, minmax(400px, 1fr))`
- `stack` : flex vertical avec `min-height: 200px` par iframe
- layouts areas : `grid-template-areas` CSS genere depuis `UiLayoutAreas`

Exemple avec areas :

```typescript
const descriptor = buildCompositeUi(resources, {
  layout: {
    areas: [
      ["filter", "main"],
      ["filter", "detail"],
    ],
    columns: [1, 3],
    rows: [2, 1],
    gap: "compact",
  },
});
```

Quand un layout areas est utilise, le renderer consulte d'abord `descriptor.areaMap`. En l'absence d'`areaMap`, il se rabat sur le nom `source` de chaque enfant comme nom d'area.

## Comportement du bus d'evenements

Le script inline gere les messages JSON-RPC 2.0 provenant des UIs enfants :

- `ui/initialize` : retourne les capacites du host et `hostContext.sharedContext`
- `ui/message` : logue un message et l'acquitte
- `ui/compose/event` : route les evenements de composition dedies en utilisant les sync rules
- `ui/update-model-context` : chemin de routage legacy, utilise aussi les sync rules

Les regles de broadcast excluent l'emetteur :

```typescript
{ from: "date:picker", event: "change", to: "*", action: "refresh" }
```

Cette regle diffuse a tous les autres iframes, pas a l'iframe source.

## Contexte partage au moment du rendu

Si le descripteur contient `sharedContext`, le renderer l'injecte dans le script :

```typescript
const descriptor = buildCompositeUi(resources, {
  layout: "stack",
  sharedContext: ["workflowId"],
});
```

La reponse d'initialisation inclut cet objet sous `hostContext.sharedContext`, et les payloads des compose-events le transmettent aussi aux iframes cibles.

## Securite et theming

Comportement actuel du renderer host dans ce repo :

- les noms d'outils sont echappes HTML avant de devenir des labels d'onglets
- `src`, `data-source`, et `data-area` sont echappes en tant qu'attributs
- le CSS de base definit des proprietes custom light et dark
- `prefers-color-scheme: dark` est inclus

Un detail d'implementation a connaitre : il existe aussi un renderer core plus ancien dans le repo, mais le package racine exporte le renderer host. Le renderer host est celui qui supporte les layouts areas et `ui/compose/event`.
