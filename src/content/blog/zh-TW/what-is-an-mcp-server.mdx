自從 Anthropic 在 2024 年底發布 **Model Context Protocol** 以來，大家都在談論 **MCP 伺服器**。但當你問「它到底是什麼？」，得到的不是一張畫滿箭頭的行銷圖表，就是一份 80 頁的規格文件。兩者都沒有真正回答問題。

以下是簡短版本：

> **MCP 伺服器是一個小型服務，透過 JSON-RPC 向 AI 智慧體暴露工具、資料和提示。** 可以把它想成 LLM 的 USB 驅動程式：每個系統（Postgres、GitHub、你自己的業務 API）一個伺服器，任何相容的客戶端——Claude、Cursor、VS Code Copilot、自製智慧體——都可以直接連接，無需撰寫特定的整合程式碼。

定義這份契約的 [Model Context Protocol](https://modelcontextprotocol.io) 由 Anthropic 於 2024 年 11 月發布，並於 2025 年 12 月移交給 Linux Foundation 旗下的 Agentic AI Foundation（AAIF）。它是目前將智慧體接入外部系統的事實標準。

以下將說明 MCP 伺服器實際暴露什麼、真實交換過程長什麼樣、它為什麼不是 REST API，以及若要部署上線需要考慮什麼。

## MCP 伺服器到底做什麼

MCP 伺服器透過 **JSON-RPC 2.0** 暴露三類原語，每類對連接的 AI 客戶端都有不同的用途。

### 工具（Tools）

工具是可呼叫的函式。MCP 伺服器為每個工具宣告名稱、描述（模型讀取描述來決定何時呼叫）以及輸入的 JSON Schema。當智慧體決定使用某個工具時，它發送一個 `tools/call` 請求；伺服器執行邏輯並回傳結果。

具體範例：一個帳單 MCP 伺服器可以暴露名為 `list_invoices` 的工具，參數為 `{ customer_id: string, limit: number }`。模型讀取描述後，判斷這是處理「顯示客戶 X 最近 5 張發票」的正確工具，並發起呼叫。伺服器查詢資料庫後回傳結構化 JSON。

工具可以讀取或寫入。MCP 規格對副作用沒有任何限制——工具可以建立記錄、發送電子郵件或觸發部署。你撰寫的描述決定了模型是否會呼叫它，因此精確性至關重要。

### 資源（Resources）

資源是可讀資料。每個資源有一個 URI 和 MIME 類型。客戶端可以呼叫 `resources/list` 來發現可用資源，再呼叫 `resources/read` 取得內容。

具體範例：一個文件 MCP 伺服器可以將 `file:///project/README.md` 暴露為 text/markdown 資源。在決定使用哪個工具之前，智慧體可以讀取 README 來了解專案結構。與工具不同，資源是被動的——它們暴露的是模型可作為上下文拉取的資料，而非可觸發的操作。

### 提示（Prompts）

提示是可複用的訊息模板。伺服器可以暴露命名模板，客戶端取得後插入對話中。具體範例：一個程式碼審查伺服器可以暴露一個 `/summarize` 提示模板，接受 diff 作為輸入並回傳結構化摘要格式。提示是目前實作中使用最少的原語，但對於標準化業務工作流程向模型的呈現方式很有價值。

## JSON-RPC 具體範例

以下是工具定義在 MCP 線格式中的樣子——即伺服器在客戶端呼叫 `tools/list` 時回傳的內容：

```json
{
  "name": "list_invoices",
  "description": "列出客戶的發票。依日期降序回傳最多 50 筆結果。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "客戶的唯一識別碼"
      },
      "limit": {
        "type": "number",
        "minimum": 1,
        "maximum": 50,
        "default": 10
      }
    },
    "required": ["customer_id"]
  }
}
```

當智慧體呼叫它時，交換過程如下：

```json
// 智慧體 → MCP 伺服器：tools/call
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "list_invoices",
    "arguments": { "customer_id": "cust_8a3f", "limit": 5 }
  }
}

// MCP 伺服器 → 智慧體：回應
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"id\":\"inv_001\",\"amount\":1200,\"status\":\"paid\"}, ...]"
      }
    ]
  }
}
```

模型拿到結構化文字內容後可以繼續推理——彙總發票、標記逾期款項，或繼續呼叫鏈中的下一個工具。

## MCP 伺服器 vs REST API：真正的差別

這個問題不斷出現：「MCP 伺服器不就只是換了個名字的 REST API 嗎？」

**簡短回答：不是。REST API 假設呼叫方已事先了解介面。MCP 伺服器在連線時自我描述，讓 AI 智慧體可以在不需要預先撰寫程式碼的情況下發現並呼叫它的能力。**

具體差異：

- **REST API** — 你硬編碼 `/api/v1/invoices`、查詢參數、認證標頭。呼叫方具備完整的先驗知識。API 若有變更，你就要重新部署客戶端。
- **MCP 伺服器** — 智慧體連接後發送 `tools/call`，收到一份機器可讀的目錄（名稱、描述、輸入 JSON Schema），再從中決定要呼叫什麼。沒有硬編碼的端點。發現過程在執行時、每個工作階段動態發生。若伺服器新增一個工具，智慧體立刻就能看到。

MCP 也是位於傳輸層（stdio 或 HTTP）之上的協定層，**不是**你現有 API 的替代品。大多數 MCP 伺服器都是轉接器：GitHub MCP 伺服器在內部呼叫 GitHub REST API；你的帳單 MCP 伺服器呼叫你的帳單 API。MCP 是面向智慧體的介面；底層資料存取維持原狀不變。

實際好處：一個 MCP 伺服器，任何 AI 客戶端，每個供應商無需特定整合程式碼。

## 誰在撰寫 MCP 伺服器

Anthropic 在[官方 MCP 伺服器儲存庫](https://github.com/modelcontextprotocol/servers)中發布了常見系統的參考伺服器——filesystem、git、GitHub、PostgreSQL、Slack、Puppeteer。這些涵蓋了最常見的開發者需求，並作為實作參考。

社群已建構了數百個更多的伺服器。[Smithery](https://smithery.ai) 和 [Glama](https://glama.ai/mcp/servers) 是兩個收錄可用 MCP 伺服器的目錄，涵蓋資料庫、雲端供應商、生產力工具、開發者服務和垂直整合。GitHub 上的 `awesome-mcp-servers` 清單是第三個起點。

企業則建構內部 MCP 伺服器，讓智慧體存取內部系統：內部 API、專有資料庫、自訂工作流程。Casys 是其中一個例子——我們建構並維護 MCP 伺服器，作為我們自己 AI 智慧體平台的組成部分。模式始終如一：將內部系統包裝在 MCP 介面之後，組織內的任何智慧體都可以呼叫它。

## 如何在生產環境建立 MCP 伺服器

讓 MCP 伺服器回應 `tools/call` 只需半天時間。讓它承受真實流量——OAuth2 認證、按客戶端限速、可觀測性、負載下的優雅關閉——則需要相當長的時間。

這些都不在 MCP 規格中。規格定義了訊息格式和三種原語。當客戶端每分鐘發送 500 個請求時如何處理、如何驗證 bearer token、如何輸出指標——這些是你自己的問題。

兩條路可以選：

1. **自行組裝。** [生產指南](/blog/mcp-server-guide)涵蓋了所有內容：從零開始的 TypeScript 實作、使用 JWT 和 OAuth2 預設的認證、限速、Prometheus 指標、OpenTelemetry 追蹤、Docker 與 systemd 部署。如果你正在建構一個需要應對真實使用的伺服器，從那裡開始。
2. **採用框架。** [`@casys/mcp-server`](/mcp-server) 是我們使用和維護的框架——在 Deno 和 Node.js 上，它原生處理認證、限速、背壓和可觀測性。你撰寫你的工具，框架負責其餘的事。

如果你計劃向智慧體暴露多個 MCP 伺服器，也可以了解一下 [MCP Gateway](/blog/mcp-gateway-guide)——一個單一入口點，多路復用並保護你的下游伺服器。

## FAQ — 關於 MCP 伺服器的常見問題

### MCP 伺服器和 REST API 有什麼差別？

REST API 假設呼叫方已事先知道哪些端點存在、需要傳遞哪些參數。MCP 伺服器在連線時透過 `tools/list` *自我描述*，讓 AI 智慧體可以在沒有硬編碼整合的情況下發現並呼叫它的能力。MCP 通常用於*包裝*現有 API，而非取代它。

### 如何建立 MCP 伺服器？

從安裝官方 SDK 開始（[TypeScript](https://github.com/modelcontextprotocol/typescript-sdk)、[Python](https://github.com/modelcontextprotocol/python-sdk)），用 JSON Schema 定義你的 `tools`，然後以 stdio 或 HTTP 模式啟動伺服器。對於生產環境（認證、限速、可觀測性），請參考我們的[逐步指南](/blog/mcp-server-guide)，或直接使用 [`@casys/mcp-server`](/mcp-server)。

### 要用什麼程式語言寫 MCP 伺服器？

任何能說 JSON-RPC 2.0 的語言都可以。Anthropic 維護 **TypeScript**、**Python**、**Java**、**Kotlin** 和 **C#** 的官方 SDK。社群也發布了 Go、Rust 和 Deno 的 SDK。由於協定與傳輸無關，任何有 HTTP 客戶端或 stdio 存取的語言都能實作 MCP 伺服器。

### 誰在維護 MCP 協定？

[Anthropic](https://anthropic.com) 於 2024 年 11 月發布 MCP，並於 2025 年 12 月將治理權移交給 **Linux Foundation** 旗下的 **Agentic AI Foundation（AAIF）**。協定現在是開放、多供應商的，並獲得 OpenAI、Google、Microsoft 及幾乎所有主流智慧體框架廠商的採用。

### 最常用的 MCP 伺服器有哪些？

Anthropic 方面：`filesystem`、`git`、`github`、`postgres`、`slack`、`puppeteer`。社群方面：Notion、Linear、Stripe、AWS、Google Drive、Sentry——可以查看 [Smithery](https://smithery.ai) 和 [Glama](https://glama.ai/mcp/servers) 這兩個目錄，收錄了數百個可用的 MCP 伺服器。
