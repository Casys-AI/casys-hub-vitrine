## 為什麼你需要關注 MCP server：2026 年 AI 架構的核心組件

2026 年，MCP 伺服器正在成為 AI 應用程式架構的核心組件。在台灣，從台積電的智慧製造到新創公司的對話式介面，開發者面對一個共同的問題：**如何讓 AI 模型安全、可靠地使用外部工具？**

[Model Context Protocol](https://modelcontextprotocol.io)（MCP）是由 Anthropic 提出、現由 Linux Foundation 託管的開放標準。它定義了 AI 模型與外部工具之間的通訊協定 — 就像 USB-C 統一了充電介面一樣，MCP 伺服器統一了工具的存取方式。無論你的用戶端是 Claude、GPT、Gemini 還是本地模型，同一個 MCP 伺服器都能直接服務。

如果你還不清楚 MCP 為什麼重要，建議先讀[這篇文章](/blog/why-mcp-protocol)。本文的重點是 **怎麼建**：從第一行程式碼到生產部署，手把手帶你完成一個 MCP 伺服器。

## MCP 伺服器的核心概念

<ExcalidrawDiagram src="mcp-server-guide/mcp-architecture" alt="MCP 架構：用戶端 ↔ 協定 ↔ 伺服器 ↔ 工具與資源" />

在動手寫程式碼之前，先搞清楚整體架構。

一個 MCP 伺服器本質上是一個 JSON-RPC 服務，暴露三類資源：

1. **Tools（工具）**— 可被 AI 模型呼叫的函式。例如查詢資料庫、發送郵件、讀取檔案。
2. **Resources（資源）**— 可被 AI 模型讀取的結構化資料。例如設定檔、使用者資訊。
3. **Prompts（提示）**— 預定義的提示範本，供 AI 模型參考使用。

通訊方式有兩種：

- **stdio**：標準輸入輸出，適合本地整合（例如 Claude Desktop、VS Code 擴充套件）
- **HTTP（Streamable HTTP + SSE）**：適合遠端部署、多用戶場景

不限定任何 AI 供應商。你建一次，到處用。

## MCP server 教學：建構你的第一個 TypeScript 範例

我們使用 [`@casys/mcp-server`](https://jsr.io/@casys/mcp-server) 框架 — 它是建立在官方 [`@modelcontextprotocol/sdk`](https://github.com/modelcontextprotocol/typescript-sdk) 之上的生產級框架，定位是 **「MCP 的 Hono」**：輕量、快速、型別安全。

### 安裝

```bash
# Deno（推薦）
deno add jsr:@casys/mcp-server

# Node.js / npm
npx jsr add @casys/mcp-server
```

`@casys/mcp-server` 同時發佈於 [JSR](https://jsr.io/@casys/mcp-server) 和 [npm](https://www.npmjs.com/package/@casys/mcp-server)，Deno 優先但完全相容 Node.js。

![JSR 上的 @casys/mcp-server 套件，100% 完美評分](/images/screenshots/jsr-casys-mcp-server.png)

### 最小可運作範例（5 行核心程式碼）

```typescript
import { McpApp } from "@casys/mcp-server";

const server = new McpApp({
  name: "my-first-server",
  version: "1.0.0",
});

server.registerTool(
  {
    name: "greet",
    description: "用指定語言問候使用者",
    inputSchema: {
      type: "object",
      properties: {
        name: { type: "string", description: "使用者名稱" },
        lang: { type: "string", enum: ["zh-TW", "en", "ja"], default: "zh-TW" },
      },
      required: ["name"],
    },
  },
  ({ name, lang }) => {
    const greetings: Record<string, string> = {
      "zh-TW": `你好，${name}！`,
      en: `Hello, ${name}!`,
      ja: `こんにちは、${name}！`,
    };
    return greetings[lang ?? "zh-TW"];
  },
);

await server.start(); // stdio 模式 — 直接接入 Claude Desktop
```

存成 `server.ts`，然後執行：

```bash
deno run --allow-all server.ts
```

這就是一個完整的伺服器。Claude Desktop 或任何 MCP 用戶端都可以直接連線。

## 工具註冊：型別安全的 Schema

伺服器的核心價值在於工具註冊。每個工具都有明確的 JSON Schema 定義，讓 AI 模型知道：這個工具做什麼、需要什麼參數、回傳什麼結果。

以下是一個較完整的範例 — 一個查詢 PostgreSQL 資料庫的工具：

```typescript
import { McpApp } from "@casys/mcp-server";

const server = new McpApp({
  name: "db-tools",
  version: "1.0.0",
  validateSchema: true, // 選配：啟用 AJV 驗證
});

// 工具 1：查詢資料庫
server.registerTool(
  {
    name: "query_database",
    description: "對 PostgreSQL 資料庫執行唯讀查詢",
    inputSchema: {
      type: "object",
      properties: {
        sql: {
          type: "string",
          description: "SELECT 查詢語句（僅允許唯讀操作）",
        },
        params: {
          type: "array",
          items: { type: "string" },
          description: "查詢參數（防止 SQL 注入）",
        },
      },
      required: ["sql"],
    },
  },
  async ({ sql, params }) => {
    // 安全檢查：僅允許 SELECT
    if (!/^\s*SELECT\b/i.test(sql)) {
      throw new Error("僅允許 SELECT 查詢");
    }
    const result = await pool.query(sql, params ?? []);
    return { rows: result.rows, rowCount: result.rowCount };
  },
);

// 工具 2：取得資料表結構
server.registerTool(
  {
    name: "describe_table",
    description: "取得指定資料表的欄位結構",
    inputSchema: {
      type: "object",
      properties: {
        table: { type: "string", description: "資料表名稱" },
      },
      required: ["table"],
    },
  },
  async ({ table }) => {
    const result = await pool.query(
      `SELECT column_name, data_type, is_nullable
       FROM information_schema.columns
       WHERE table_name = $1
       ORDER BY ordinal_position`,
      [table],
    );
    return result.rows;
  },
);

await server.start();
```

Schema 驗證是選配功能，透過 `validateSchema: true` 啟用。啟用後，`@casys/mcp-server` 會在每次呼叫前用 [AJV](https://ajv.js.org/) 驗證參數。不合規的請求直接被拒絕，不會到達你的處理邏輯 — 這在生產環境中至關重要。

## 中介軟體：可組合的洋蔥模型

<ExcalidrawDiagram src="mcp-server-guide/mcp-server-middleware-pipeline" alt="@casys/mcp-server 中介軟體管線：Rate Limit → Auth → Validation → Backpressure → Handler" />

`@casys/mcp-server` 的中介軟體架構和 Hono、Koa 一樣：**洋蔥模型（onion model）**。你可以在請求的前後插入任意邏輯。

```typescript
import type { Middleware } from "@casys/mcp-server";

// 計時中介軟體
const timing: Middleware = async (ctx, next) => {
  const start = performance.now();
  const result = await next();
  const duration = (performance.now() - start).toFixed(0);
  console.log(`[${ctx.toolName}] ${duration}ms`);
  return result;
};

// 稽核日誌中介軟體
const audit: Middleware = async (ctx, next) => {
  console.log(`[AUDIT] tool=${ctx.toolName} user=${ctx.auth?.sub ?? "anon"}`);
  return next();
};

server.use(timing);
server.use(audit);
```

內建的管線順序是：

```
速率限制 → 驗證 → 自訂中介軟體 → 範圍檢查 → Schema 驗證 → 背壓控制 → 處理器
```

你只需要寫自訂邏輯，安全與效能相關的層由框架處理。

## HTTP 模式：遠端部署的 MCP 伺服器

stdio 適合本地開發。但如果你要服務多個用戶端、部署在雲端，就需要 HTTP 模式。

```typescript
import {
  McpApp,
  createAuth0AuthProvider,
} from "@casys/mcp-server";

const server = new McpApp({
  name: "production-api",
  version: "1.0.0",
  maxConcurrent: 20,
  backpressureStrategy: "queue", // "queue" | "drop" | "error"
  validateSchema: true,
  rateLimit: { maxRequests: 100, windowMs: 60_000 },
  auth: {
    provider: createAuth0AuthProvider({
      domain: "my-tenant.auth0.com",
      audience: "https://mcp.example.com",
      resource: "https://mcp.example.com",
      scopesSupported: ["read", "write", "admin"],
    }),
  },
});

// 註冊工具...（同上）

await server.startHttp({
  port: 3000,
  requireAuth: true,        // 未設定 auth provider 時直接失敗
  corsOrigins: ["https://app.example.com"],
  maxBodyBytes: 1_048_576,   // 1 MB
  ipRateLimit: { maxRequests: 60, windowMs: 60_000 },
});

// 路由說明：
// POST /mcp       → JSON-RPC（tools/call, tools/list 等）
// GET  /mcp       → SSE 串流（伺服器 → 用戶端通知）
// GET  /health    → { status: "ok" }
// GET  /metrics   → Prometheus 格式指標
```

### 安全預設

`@casys/mcp-server` 在 HTTP 模式下的設計哲學是 **安全優先（secure by default）**：

- `requireAuth: true` — 沒有設定驗證供應商就無法啟動
- `maxBodyBytes` 預設 1 MB — 防止記憶體耗盡攻擊
- `corsOrigins` 預設 `"*"` 但會在日誌中印出警告 — 生產環境必須設定白名單
- `ipRateLimit` — 每個 IP 獨立的速率限制，防止暴力攻擊
- JWT 權杖快取（SHA-256 雜湊 → AuthInfo，TTL = min(權杖過期時間, 5 分鐘)）— 避免每次呼叫都去 JWKS 端點驗證

### MCP Apps：當客戶端支援時返回 UI

從 v0.14 開始，框架支援 [MCP Apps 能力協商](https://github.com/nichochar/mcp-apps-spec)（規範 `2026-01-26`）。如果客戶端在 `initialize` 握手中宣布支援互動式 UI，你的處理程式可以返回 UI 資源而非純文字：

```typescript
server.registerTool(
  {
    name: "dashboard",
    description: "Show the monitoring dashboard",
    inputSchema: { type: "object", properties: {} },
  },
  async (_args, { server }) => {
    if (server.getClientMcpAppsCapability()) {
      return {
        content: [{ type: "text", text: "Dashboard loaded." }],
        _meta: { ui: { resourceUri: "app://dashboard/main" } },
      };
    }
    return "Dashboard not available — client does not support MCP Apps.";
  },
);
```

框架匯出常數 `MCP_APP_MIME_TYPE`、`MCP_APPS_EXTENSION_ID` 和 `MCP_APPS_PROTOCOL_VERSION` 以便整合。

## OAuth2 / JWT 驗證

生產環境中必須有驗證。`@casys/mcp-server` 內建四種 OIDC 預設：Google、Auth0、GitHub Actions（CI 工作流程 OIDC token，非使用者帳號）、以及通用 OIDC（Keycloak、Okta 等）。

```typescript
import {
  createGoogleAuthProvider,   // Google OIDC
  createAuth0AuthProvider,    // Auth0
  createGitHubAuthProvider,   // GitHub Actions OIDC
  createOIDCAuthProvider,     // 通用 OIDC（Keycloak、Okta 等）
} from "@casys/mcp-server";
```

也可以用 `JwtAuthProvider` 自訂：

```typescript
import { JwtAuthProvider } from "@casys/mcp-server";

const provider = new JwtAuthProvider({
  issuer: "https://my-idp.example.com",
  audience: "https://my-mcp.example.com",
  resource: "https://my-mcp.example.com",
  authorizationServers: ["https://my-idp.example.com"],
});
```

工具可以設定 `requiredScopes` 來限制存取範圍：

```typescript
server.registerTool(
  {
    name: "delete_record",
    description: "刪除一筆記錄",
    inputSchema: { /* ... */ },
    requiredScopes: ["admin", "write"], // 必須同時擁有兩個 scope
  },
  async ({ id }) => { /* ... */ },
);
```

### YAML 設定檔

對於二進位發行版，使用者不需要改程式碼 — 用 YAML 設定即可：

```yaml
# mcp-server.yaml
auth:
  provider: auth0
  audience: https://mcp.example.com
  resource: https://mcp.example.com
  domain: my-tenant.auth0.com
  scopesSupported: [read, write, admin]
```

環境變數覆蓋優先順序：`程式碼 > 環境變數 > YAML`

```bash
MCP_AUTH_AUDIENCE=https://prod.example.com ./my-server --http --port 3000
```

## 可觀測性：OpenTelemetry 與 Prometheus

生產級部署需要可觀測性。`@casys/mcp-server` 內建兩套機制：

### OpenTelemetry 追蹤

每個工具呼叫自動產生一個 OTel span：

```
mcp.tool.call {
  tool.name: "query_database"
  server.name: "production-api"
  transport: "http"
  session.id: "abc123"
  duration_ms: 42
  success: true
}
```

啟用方式（Deno）：

```bash
OTEL_DENO=true deno run --unstable-otel --allow-all server.ts
```

### Prometheus 指標

`GET /metrics` 端點暴露：

- **Counters（計數器）**：工具呼叫次數、驗證事件、速率限制觸發次數
- **Histogram（直方圖）**：工具呼叫延遲分佈
- **Gauges（即時值）**：活躍請求數、佇列長度、SSE 用戶端數、伺服器運行時間

```bash
curl http://localhost:3000/metrics
# mcp_tool_calls_total{tool="query_database",status="success"} 1247
# mcp_tool_call_duration_seconds_bucket{tool="query_database",le="0.1"} 1189
# mcp_active_requests 3
```

接上 Grafana，你就有完整的監控儀表板。

## 連接 AI 用戶端

任何實作 MCP 協定的用戶端都可以連線。stdio 模式的設定格式（適用 Claude Desktop、Claude Code 等）：

```json
{
  "mcpServers": {
    "my-db-tools": {
      "command": "deno",
      "args": ["run", "--allow-all", "/path/to/server.ts"]
    }
  }
}
```

HTTP 遠端模式則改用 `"url": "https://mcp.example.com"` 並附上 `Authorization` header。

## 使用內建 inspector 測試

框架匯出 `launchInspector`，可直接從伺服器啟動 MCP inspector——無需安裝外部套件：

```typescript
import { launchInspector } from "@casys/mcp-server";

if (Deno.args.includes("--inspect")) {
  await launchInspector("deno", ["run", "--allow-all", "server.ts"]);
}
```

使用 `--inspect` 啟動伺服器，即可在 `localhost:6274` 開啟除錯 Web UI——工具呼叫、資源瀏覽、原始 JSON-RPC 回應。

## 框架對比：三種方式建構 MCP 伺服器

<ExcalidrawDiagram src="_shared/mcp-ecosystem-2026" alt="2026 年 MCP 生態系統" />

| 特性 | 從零自建 | @modelcontextprotocol/sdk | @casys/mcp-server |
| --- | :---: | :---: | :---: |
| MCP 協定相容性 | 需自行實作 | 完整 | 完整（基於官方 SDK） |
| 並行控制 | 無 | 無 | 三種背壓策略 |
| 中介軟體管線 | 無 | 無 | 洋蔥模型（Hono 風格） |

| 速率限制 | 需自行實作 | 無 | 按 IP（HTTP）和可配置 key（工具、會話、自訂） |
| Schema 驗證 | 需自行實作 | 無 | AJV（透過 `validateSchema` 啟用） |
| Streamable HTTP + SSE | 需自行實作 | 需手動設定 | 內建 Session 管理 |
| OpenTelemetry 追蹤 | 需自行整合 | 無 | 自動 span |
| Prometheus 指標 | 需自行實作 | 無 | `/metrics` 端點 |
| MCP Apps（UI 資源） | 需自行實作 | 需手動處理 | `registerResource()` + `ui://` |
| Sampling 橋接 | 需自行實作 | 無 | 雙向 LLM 委派 |
| YAML + 環境變數設定 | 無 | 無 | 內建 |
| 上手時間 | 數天～數週 | 數小時 | 數分鐘 |

**官方 SDK** 給你協定。**@casys/mcp-server** 給你生產堆疊。

如果你的需求是「快速原型驗證，只要能跑」，官方 SDK 夠用。但如果你要部署到生產環境，面對多租戶、需要驗證與監控，從 SDK 開始你會重新發明輪子。`@casys/mcp-server` 把這些輪子都預製好了。

## MCP 伺服器生產部署：完整 TypeScript 範例

<ExcalidrawDiagram src="mcp-server-guide/mcp-server-deployment" alt="生產部署架構：用戶端 → Caddy (TLS) → MCP Server → PostgreSQL，搭配 Prometheus 與 OpenTelemetry 監控" />

以下是一個生產級部署的完整範例，包含驗證、可觀測性與背壓控制：

```typescript
import {
  McpApp,
  createOIDCAuthProvider,
} from "@casys/mcp-server";
import type { Middleware } from "@casys/mcp-server";

// — 中介軟體 —

const requestLogger: Middleware = async (ctx, next) => {
  const start = performance.now();
  try {
    const result = await next();
    console.log(
      JSON.stringify({
        tool: ctx.toolName,
        user: ctx.auth?.sub,
        duration_ms: Math.round(performance.now() - start),
        status: "ok",
      }),
    );
    return result;
  } catch (err) {
    console.error(
      JSON.stringify({
        tool: ctx.toolName,
        user: ctx.auth?.sub,
        duration_ms: Math.round(performance.now() - start),
        status: "error",
        error: (err as Error).message,
      }),
    );
    throw err;
  }
};

// — 伺服器 —

const server = new McpApp({
  name: "tw-manufacturing-api",
  version: "1.0.0",
  maxConcurrent: 50,
  backpressureStrategy: "queue",
  validateSchema: true,
  rateLimit: { maxRequests: 200, windowMs: 60_000 },
  auth: {
    provider: createOIDCAuthProvider({
      issuer: "https://keycloak.internal.example.com/realms/mfg",
      audience: "https://mcp.factory.example.com",
      resource: "https://mcp.factory.example.com",
    }),
  },
});

server.use(requestLogger);

// — 工具 —

server.registerTool(
  {
    name: "get_production_metrics",
    description: "取得指定產線的即時生產指標",
    inputSchema: {
      type: "object",
      properties: {
        lineId: { type: "string", description: "產線編號" },
        metric: {
          type: "string",
          enum: ["yield", "throughput", "defect_rate", "oee"],
          description: "指標類型",
        },
        timeRange: {
          type: "string",
          enum: ["1h", "8h", "24h", "7d"],
          default: "8h",
        },
      },
      required: ["lineId", "metric"],
    },
    requiredScopes: ["production:read"],
  },
  async ({ lineId, metric, timeRange }) => {
    // 連接工廠 MES 系統...
    return {
      lineId,
      metric,
      value: 98.7,
      unit: metric === "defect_rate" ? "ppm" : "%",
      timestamp: new Date().toISOString(),
    };
  },
);

server.registerTool(
  {
    name: "create_quality_alert",
    description: "建立品質異常警報",
    inputSchema: {
      type: "object",
      properties: {
        lineId: { type: "string" },
        severity: { type: "string", enum: ["low", "medium", "high", "critical"] },
        description: { type: "string" },
      },
      required: ["lineId", "severity", "description"],
    },
    requiredScopes: ["production:write", "quality:alert"],
  },
  async ({ lineId, severity, description }) => {
    // 發送到品質管理系統...
    return { alertId: crypto.randomUUID(), status: "created" };
  },
);

// — 啟動 —

await server.startHttp({
  port: Number(Deno.env.get("PORT") ?? 3000),
  requireAuth: true,
  corsOrigins: [
    "https://dashboard.factory.example.com",
  ],
  maxBodyBytes: 1_048_576,
  ipRateLimit: { maxRequests: 120, windowMs: 60_000 },
});

console.log("MCP 伺服器已啟動，等待連線...");
```

### 搭配 systemd 部署

```ini
# /etc/systemd/system/mcp-factory.service
[Unit]
Description=Factory MCP Server
After=network.target

[Service]
Type=simple
User=mcp
WorkingDirectory=/opt/mcp-factory
ExecStart=/usr/bin/deno run --allow-net --allow-env --allow-read server.ts
Restart=always
RestartSec=5
Environment=PORT=3000
Environment=OTEL_DENO=true
Environment=DENO_UNSTABLE_OTEL=true

[Install]
WantedBy=multi-user.target
```

### 搭配反向代理（Caddy）

```
mcp.factory.example.com {
    reverse_proxy localhost:3000
}
```

TLS 由 Caddy 自動處理，伺服器只需要監聽 localhost。

## 路由的下一步：不只是工具呼叫

建好伺服器只是第一步。當你的工具數量成長到數十、數百個時，一個新的問題浮現：**AI 模型怎麼知道該用哪個工具？**

這就是[路由問題](/blog/routing-agents-without-new-protocol) — 當 MCP 生態系統中有超過 17,000 個工具時，沒有任何上下文窗口能裝下所有工具的 schema。你需要一個專用的路由層，在 AI 模型開始推理之前，先決定給它看哪些工具。

`@casys/mcp-server` 的 sampling 橋接功能就是為此設計的：伺服器可以反向請求宿主 LLM 進行推理，實現雙向溝通。這不是一個備用方案 — 這是一個編排原語。

### 使用 @casys/mcp-compose 組合 UI

路由解決了工具的問題。但當每個 MCP 伺服器返回自己的 UI（透過 MCP Apps）時，代理會面對碎片化的視圖。[`@casys/mcp-compose`](https://jsr.io/@casys/mcp-compose) 編排這些 UI：統一儀表板、面板間狀態同步和宣告式事件路由。這不是請求路由——而是客戶端視圖組合。

## 選擇 MCP 框架：Python 還是 TypeScript？

建構生產級 MCP 伺服器的兩個主要框架是 [FastMCP](https://gofastmcp.com/)（Python）和 `@casys/mcp-server`（TypeScript）。兩者解決的是同一個問題 — 填補原生 SDK 和可部署服務之間的落差 — 但面向不同的生態系。

### FastMCP

FastMCP 是 MCP 的 Python 參考框架。它從社群項目起步，被納入官方 Python SDK，之後以獨立框架重新推出。FastMCP 3.0 圍繞三個原語 — 元件、提供者、轉換器 — 強調動態工具組合。如果你的技術堆疊是 Python，這是自然的選擇：可直接使用 NumPy、pandas、Hugging Face，並透過 `@mcp.tool()` 快速建立原型。

### @casys/mcp-server

`@casys/mcp-server` 是 TypeScript 的對應方案。它面向希望在伺服器、客戶端、測試和 IDE 工具鏈間保持同一語言的團隊。API 靈感來自 Hono（洋蔥模型中介軟體管線），生產防護 — Prometheus `/metrics`、OpenTelemetry span、速率限制、OAuth2 驗證 — 內建於框架而非選配模組。

[![AI 代理透過 @casys/mcp-erpnext 自動建立 ERPNext 發票 — 觀看示範](/images/screenshots/erpnext-mcp-demo.jpg)](https://youtu.be/gG-nFCrzfPU)

### 功能對比表

| 特性 | FastMCP（Python） | @casys/mcp-server（TypeScript） |
|---|---|---|
| 執行環境 | CPython、uv | Deno + Node.js |
| 驗證（OAuth2/JWT） | 逐元件、CIMD、Azure OBO | 4 種 OIDC 預設、逐工具 scope |
| 中介軟體管線 | Providers + transforms | 洋蔥模型（Hono 風格） |
| 速率限制 | 手動 / 第三方 | 按 IP + 可配置 key |
| 可觀測性 | 需手動串接 | Prometheus + OpenTelemetry 內建 |
| 生態系 | Python ML 函式庫 | mcp-bridge、mcp-compose、mcp-erpnext |

### 選擇取決於你的技術堆疊

如果你的後端是 Python 並使用 ML 函式庫，FastMCP 是正確的選擇。如果你的技術堆疊是 TypeScript，希望從 MCP 伺服器到 IDE 擴充套件都使用同一語言，`@casys/mcp-server` 涵蓋相同的生產範圍。兩個項目都是開源的，都建構在官方 SDK 之上，為不同的社群解決同一個問題。

## 開始使用

官方 MCP SDK 足以應付原型或本機 `stdio` 工具。對於需要驗證、速率限制和可觀測性的生產服務，`@casys/mcp-server` 省去為每個新伺服器重新實作這些層的工作。

```bash
# Deno
deno add jsr:@casys/mcp-server

# Node.js (npm)
npm install @casys/mcp-server
```

框架是開源的，發佈於 [JSR](https://jsr.io/@casys/mcp-server) 和 [npm](https://www.npmjs.com/package/@casys/mcp-server)，建構在官方 SDK 之上。原始碼在 [GitHub](https://github.com/Casys-AI/mcp-server)。
