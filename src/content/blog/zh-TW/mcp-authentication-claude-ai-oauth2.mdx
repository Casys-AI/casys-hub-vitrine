# 將你的 MCP 伺服器連接到 claude.ai

> 讀完這份指南，你的遠端 MCP 伺服器將可以從 claude.ai 存取，每個請求都會帶有已驗證的使用者身份，你也會清楚知道哪些 Auth0 設定真的重要、哪些不重要。

---

## 這份指南涵蓋什麼

如果你在建構遠端 MCP 伺服器，遲早會想讓它在 [claude.ai](https://claude.ai) 上運作——也就是網頁版，而不只是桌面用戶端。網頁版才是大多數使用者的所在地，也是非技術協作者不需要安裝任何東西就能使用你工具的地方。

整合方式*並不是*「貼上這個 Bearer 令牌」。claude.ai 的 MCP 用戶端只接受 OAuth2：一個完整的授權碼流程，包含探索、重新導向 URI 以及符合 RFC 9068 的 JWT。這份指南會帶你從一個可以在本地運作的 MCP 伺服器，走到一個 claude.ai 能夠代表真實使用者與之連線的生產環境設定。

**你將學會**：

- 為什麼 claude.ai 會拒絕靜態 Bearer 令牌，以及它實際期待什麼
- 如何從零開始設定一個 Auth0 租戶 + API + 應用程式（如果你知道要撥哪些開關，10 分鐘就夠）
- 哪兩個 Auth0 設定不直觀，以及忽略它們會讓你浪費幾個小時除錯
- 如何在你的 MCP 伺服器中間件裡驗證 RFC 9068 JWT（附完整 TypeScript 範例）
- 你會碰到的五種失敗情境，以及如何在不到一分鐘內各自修復

**你需要準備**：

- 一個已經可以在 MCP Inspector 或 Claude Desktop 上透過靜態令牌運作的遠端 MCP 伺服器
- 一個 Auth0 帳戶（免費方案就夠用了）
- 第一次 30 分鐘，之後每個 MCP 伺服器 5 分鐘

這篇文章是 [iframe 捲動問題](/blog/mcp-apps-mobile-ux-patterns) 的配套文章——兩者共享同一個主題：**當你的用戶端是別人的應用程式，不要與它對抗，去學它的規則。**

## 為什麼靜態令牌行不通

在開始設定之前，先理解*為什麼*顯而易見的方法會失敗——失敗模式並不直觀，錯誤訊息也沒什麼用。

一個典型的起點：一個 [Deno Deploy](https://deno.com/deploy) 伺服器（我們整篇指南都會用 [mcp-einvoice](https://github.com/casysai/mcp-einvoice) 作為實際範例），透過 Streamable HTTP 在 `https://mcp-einvoice.casys.deno.net/mcp` 發布工具，用靜態 Bearer 令牌做標頭驗證。這個設定在 MCP Inspector、`curl`，以及透過 `claude_desktop_config.json` 預先填入 Authorization 標頭的 Claude Desktop 裡都能運作。到目前為止，一切正常。

然後你去 claude.ai → 設定 → 整合 → 新增 MCP 伺服器，貼上 URL，UI 只問你一件事：「伺服器 URL」。沒有令牌欄位。沒有標頭欄位。沒有「進階選項」。只有 URL。

你按下連線。claude.ai 打到你的伺服器。你的伺服器回傳 `401 Unauthorized`。然後 claude.ai 做了一件你可能沒想到的事：它去找 `/.well-known/oauth-protected-resource`。

<ExcalidrawDiagram src="mcp-auth-claude-ai/expectation-vs-reality" alt="Expectation: claude.ai sends Authorization: Bearer to your MCP server. Reality: claude.ai does OAuth2 discovery, authenticates against Auth0, gets an RS256 JWT, then talks to your MCP server." />

那個 URL 是 [MCP 授權規範](https://modelcontextprotocol.io/specification/draft/basic/authorization) 的實際運作。當 claude.ai 打到一個 MCP 端點並得到 401 時，它期望伺服器回應一個**受保護資源元資料**文件，指向一個 OAuth2 授權伺服器。靜態 Bearer 驗證不在規範裡。它從來就不在規範裡。桌面用戶端讓你用標頭矇混過去——網頁用戶端不行。

claude.ai 沒有壞掉。Bearer 令牌模型只是不符合它的形狀。這份指南其餘的部分，就是關於如何交付正確的形狀。

## 兩段話解釋規範

以下是 MCP 授權的故事，去掉所有術語。

**MCP 伺服器是受保護資源（Protected Resource）。** 它不發行令牌。它不驗證密碼。它只接受由授權伺服器簽發的 RS256 JWT，而該授權伺服器的身份由 MCP 伺服器在一個已知 URL 上公開。把它想成你的 API 與 Auth0 之間、或你的 API 與 Okta 之間的同一種分離。API 永遠看不到密碼。

**MCP 用戶端（claude.ai）是 OAuth2 用戶端。** 它從你的受保護資源元資料中探索授權伺服器，引導使用者完成授權碼流程，用授權碼交換 Bearer JWT，然後在每個 MCP 請求的 `Authorization` 標頭裡帶上那個 JWT。使用者**只需登入一次**，claude.ai 快取令牌，之後的會話就是每個使用者各自獨立的。

在繼續之前，有兩個結論你應該先內化：

1. **沒有共享憑證。** 你的 MCP 伺服器的每個使用者都有自己的身份。你的伺服器在不同使用者之間永遠不會看到相同的令牌。撤銷是免費的——在 Auth0 裡停用使用者，他們的令牌在下次刷新時就失效了。
2. **你不是在建構驗證伺服器。** 你只是在*指向*一個驗證伺服器。這意味著你的工作比「實作 OAuth2」小得多——它是「正確設定 Auth0，並加一個小型元資料端點」。

當我在第一天意識到這件事時，我的反應是惱怒。到了第三天，它翻轉成了「哦——這實際上才是正確的設計」。在我們進入 YAML 之前，讓我解釋為什麼。

## 為什麼這是正確的選擇

老實說：我的第一直覺是「這是 Anthropic 在耍威風」。靜態 Bearer 對 90% 的 API 整合來說都夠用，強制每個業餘 MCP 伺服器都用 OAuth2，看起來像是被貨物崇拜的企業級安全。

我錯了。以下是理由，按強度排序：

**使用者同意對 MCP 工具來說不是選項。** 一個 MCP 工具可以讀取我的發票。它可以在我的 ERP 裡建立使用者。它可以刪除文件。當 claude.ai 要我連線一個伺服器時，我需要看到一個同意畫面，告訴我*確切地*我授予了哪些範圍，來自一個我信任的網域，用我自己的憑證。「貼上這個令牌」的欄位完全不提供這些。第一次惡意的 MCP 工具透過設定錯誤的靜態金鑰外洩了某人的資料，整個生態系都要承擔聲譽損失。強制使用 OAuth2 在設計上就把那個失敗模式推到了零。

**每個使用者的身份也不是選項。** 如果同一家公司的兩個人連線了同一個 MCP 伺服器，他們**不應該**共享一個令牌。他們應該各自有自己的。Auth0 輕鬆做到這件事。靜態 API 金鑰做不到。一旦你需要多租戶、每個使用者的稽核日誌、每個使用者的速率限制，或每個使用者的資料範圍隔離，靜態金鑰就垮了，你最終還是在拙劣地重新發明 OAuth2。

**撤銷是一等操作。** 當筆電遺失、前員工離職，或使用者輪換憑證時，OAuth2 給你一個按鈕可以按。靜態 Bearer 要求你去追查每個貼過金鑰的地方。

**範圍（Scopes）。** 一個設計良好的 MCP 伺服器有唯讀範圍、寫入範圍、管理員範圍。OAuth2 讓這成為一等概念。使用者在同意畫面上看到，確切地，他們授予 claude.ai 哪些能力。用靜態金鑰，就是全有或全無。

所以：是的，這需要更多工作。是的，你必須設定 Auth0 或等價的東西。但這跟「更多工作」是同樣的，就像讓我們從明文儲存密碼改成用 bcrypt。這不是官僚障礙，這是一個在生產系統中支撐了十五年的模式，現在被應用到 AI 工具層。對抗它，你會輸。採用它，你會睡得更好。

現在讓我們讓它運作起來。

## 流程的形狀

在深入 Auth0 之前，先看完整的流程。把它燒進你的腦袋——這會幫你省下兩個小時迷茫地在不同 console 分頁之間切換。

<ExcalidrawDiagram src="mcp-auth-claude-ai/oauth2-flow" alt="MCP OAuth2 flow on claude.ai: claude.ai discovers the protected resource metadata, redirects to Auth0, user logs in and consents, Auth0 redirects back to claude.ai/api/mcp/auth_callback, claude.ai exchanges the code for an RS256 JWT, claude.ai calls the MCP server with Bearer JWT." />

十個步驟，四個參與者。你精確地控制其中兩個：你的 MCP 伺服器（步驟 1、2、9、10）和 Auth0（步驟 4、5、7、8）。claude.ai 和使用者的瀏覽器自己完成其餘的部分。

關於這張圖最重要的一件事：**claude.ai 的重新導向 URI 是固定的。** 它是 `https://claude.ai/api/mcp/auth_callback`。你不能選擇它。你不能設定它。你一字不差地把它註冊到 Auth0 用戶端，否則什麼都不會運作。一旦你試圖聰明地把人重新導向到別的地方，Auth0 的回調檢查就失敗了，瀏覽器會落在一個沒人能幫你解決的錯誤頁面。

好。這是設定步驟。

## Auth0 十分鐘搞定（重要的那些部分）

我用 [Auth0](https://auth0.com) 是因為一旦你知道要撥哪些開關，它就能開箱即用地正確處理 MCP。你也可以用 Keycloak、Authentik、Clerk、Stytch，或任何符合 RFC 9068 的授權伺服器。下面的*概念*都是一樣的。Auth0 只是針對 MCP 特別有最好的文件，透過他們的 [Auth for AI](https://auth0.com/ai/docs/mcp/intro/overview) 文件。

### 1. 租戶（Tenant）

建立一個租戶。我用 `casys.eu.auth0.com`。任何區域都可以——選一個靠近你使用者的，因為這在登入延遲的關鍵路徑上。

### 2. API（你的 MCP 伺服器作為受保護資源）

這是 Auth0 中從 OAuth2 角度代表你 MCP 伺服器的物件。建立它：

- **名稱**：`MCP eInvoice`（你想取什麼都行，這只是標籤）
- **識別符（Audience）**：`https://mcp-einvoice.casys.deno.net/` — **你的 MCP 伺服器的確切 URL，帶尾部斜線。** Auth0 用這個字串作為每個發行 JWT 的 `aud` 聲明，你的 MCP 伺服器之後會檢查傳入令牌的 `aud` 是否與自己的 URL 相符。多一個或少一個斜線就是 401。

儲存。Auth0 會帶你到 API 的設定頁面——讓它開著，下一步你會需要它。

### 3. 令牌格式：切換到 RFC 9068

MCP 授權規範明確要求 [RFC 9068](https://www.rfc-editor.org/rfc/rfc9068.html)（「OAuth 2.0 存取令牌的 JWT 設定檔」）。MCP 用戶端函式庫——因此 claude.ai——會針對那個設定檔進行驗證。如果你的 `aud`、`iss`、`exp`、`iat`、`scope` 和 `jti` 聲明不完全符合，用戶端就拒絕令牌，你在成功的 OAuth2 往返*之後*得到 401。這是整個堆疊裡最令人困惑的失敗模式：OAuth 流程在網路分頁裡看起來沒問題，JWT 在 jwt.io 上解碼得很乾淨，但你的伺服器還是說不行。

Auth0 的 API 預設是舊版「Auth0」令牌格式。你必須切換它。設定在 **APIs → 你的 API → Settings → JSON Web Token (JWT) Profile**，只有兩個選項：

<figure class="blog-screenshot">
  <img src="/images/blog/mcp-auth/access-token-settings.png" alt="Auth0 dashboard — API Settings → Access Token Settings. JSON Web Token (JWT) Profile dropdown set to RFC 9068. Signing Algorithm RS256. JSON Web Encryption disabled." loading="lazy" />
  <figcaption>APIs → 你的 API → Settings → Access Token Settings。Profile 選 <code>RFC 9068</code>，Signing Algorithm 選 <code>RS256</code>。儲存。</figcaption>
</figure>

- **JWT Profile** = `RFC 9068`。預設的「Auth0」格式產生的 JWT 看起來是對的，但缺少 MCP 驗證器期待的嚴格 RFC 9068 聲明形狀。RFC 9068 修正了這個問題。
- **Signing Algorithm** = `RS256`。不是 HS256。HS256 需要 Auth0 和你的伺服器之間有共享密鑰，而這正是我們試圖避免的。RS256 是公/私鑰——Auth0 用私鑰簽名，你的伺服器用 `https://<tenant>/.well-known/jwks.json` 的公開 JWKS 驗證，環境裡不需要秘密。
- **JSON Web Encryption** = 關閉。你不需要它。令牌已經被簽名並透過 HTTPS 傳輸。

較舊的 Auth0 文件（以及儀表板的早期版本）參照一個 `token_dialect` 欄位，值像是 `access_token_authz` 和 `rfc9068_profile_authz`，透過 Management API 設定。那是舊版的做法。儀表板現在直接以「JWT Profile」的形式公開選擇，Auth0 儲存的值是 `rfc9068_profile`。如果你發現自己在透過 `auth0 api` 做修補，你在讀舊文件。

儲存。你的 API 下一個發行的令牌將會符合 RFC 9068。

### 4. 用戶端（claude.ai 作為 OAuth2 用戶端）

在 Auth0 裡，一個「應用程式」（他們指的是 OAuth2 用戶端）代表 claude.ai：

- **類型**：Regular Web Application。
- **名稱**：`Claude.ai`。
- **Allowed Callback URLs**：`https://claude.ai/api/mcp/auth_callback` — **就是這個，一字不差，不帶尾部斜線，不帶查詢字串**。這是 claude.ai 固定的重新導向 URI，你不能協商它。
- **Grant Types**：Authorization Code。Refresh Token。
- **Connections**：至少啟用一個。我用 Username-Password-Authentication（資料庫）和 google-oauth2（社交）。使用者在登入時選擇一個。

儲存。Auth0 給你一個 `client_id`。複製它——使用者會在流程中的瀏覽器 URL 裡看到它，但你的 MCP 伺服器程式碼裡不需要它。

### 5. 那個看不見的開關

這是讓我找到時笑出聲的那個。

<figure class="blog-screenshot">
  <img src="/images/blog/mcp-auth/resource-parameter-toggle.png" alt="Auth0 dashboard — Settings → Advanced → Resource Parameter Compatibility Profile toggle, enabled. Profile that determines how the identity of the protected resource is specified in OAuth endpoints." loading="lazy" />
  <figcaption>Tenant Settings → Advanced → Resource Parameter Compatibility Profile。預設關閉。打開它，否則 claude.ai 的 <code>/authorize</code> 請求會以 <code>invalid_request</code> 失敗。</figcaption>
</figure>

Auth0 有一個租戶層級的設定叫做 **Resource Parameter Compatibility Profile**，埋在 Settings → Advanced 裡。它**預設是關閉的**。當它關閉時，Auth0 不接受 `/authorize` 和 `/oauth/token` 上的 `resource` 參數——而那個參數是 claude.ai 用來告訴 Auth0「我在為這個特定的 MCP 伺服器請求令牌」的。

沒有它，claude.ai 的授權請求就被拒絕為 `invalid_request`，你看到一個通用錯誤頁面，錯誤訊息裡既沒有提到「resource」也沒有提到「MCP」。我只是因為 Auth0 for AI 文件裡的一句不起眼的話才找到它。花了兩個小時。

**打開它。一次。每個租戶。** 不需要 CLI 操作，不需要修補——就翻轉那個開關然後儲存。

到這裡 Auth0 就準備好了。現在來教你的 MCP 伺服器如何成為受保護資源。

## MCP 伺服器端：兩個端點

你的 MCP 伺服器需要在現有基礎上新增兩件事：

1. 在 `/.well-known/oauth-protected-resource` 的**受保護資源元資料**端點。
2. MCP 路由上的 **JWT 驗證**中間件。

其他所有東西保持不變。

### 受保護資源元資料

claude.ai 首先（在最初的 401 之後）打這個 URL，以探索在哪裡進行身份驗證。規範是 [RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728)。最小化的 Payload：

```typescript
// src/routes/protected-resource.ts
export function protectedResourceMetadata() {
  return Response.json({
    resource: "https://mcp-einvoice.casys.deno.net/",
    authorization_servers: [
      "https://casys.eu.auth0.com/"
    ],
    scopes_supported: [
      "mcp:read",
      "mcp:write"
    ],
    bearer_methods_supported: ["header"],
    resource_documentation: "https://casys.ai/engine/mcp-einvoice"
  });
}
```

`resource` 欄位必須與你在 Auth0 設定的 Audience 完全相符。`authorization_servers` 欄位告訴 claude.ai OAuth2 端點在哪裡——Auth0 從那個根路徑發布它自己的 `/.well-known/openid-configuration`，所以 claude.ai 可以探索 `/authorize` 和 `/oauth/token`，不需要你把它們硬編碼進去。

在 `/.well-known/oauth-protected-resource` 上提供那個 JSON。探索這側就完成了。

### JWT 驗證中間件

對每個 MCP 請求，驗證 JWT：

```typescript
// src/middleware/auth.ts
import { jwtVerify, createRemoteJWKSet } from "jose";

const JWKS = createRemoteJWKSet(
  new URL("https://casys.eu.auth0.com/.well-known/jwks.json")
);

export async function authenticate(req: Request) {
  const auth = req.headers.get("authorization");
  if (!auth?.startsWith("Bearer ")) {
    return { error: unauthorized("missing_token") };
  }
  const token = auth.slice(7);

  try {
    const { payload } = await jwtVerify(token, JWKS, {
      issuer: "https://casys.eu.auth0.com/",
      audience: "https://mcp-einvoice.casys.deno.net/",
    });
    return { user: payload };
  } catch (e) {
    return { error: unauthorized("invalid_token") };
  }
}

function unauthorized(code: string) {
  return new Response(null, {
    status: 401,
    headers: {
      "WWW-Authenticate": `Bearer realm="mcp-einvoice", error="${code}", resource_metadata="https://mcp-einvoice.casys.deno.net/.well-known/oauth-protected-resource"`
    }
  });
}
```

三個值得停下來思考的細節：

1. **JWKS 有快取。** `createRemoteJWKSet` 一次性取得 Auth0 的公鑰並快取。不要每個請求都重新取得——你會觸發速率限制，並給每個呼叫增加 200ms。
2. **`issuer` 和 `audience` 是硬編碼的檢查。** 這些是你為 RFC 9068 令牌付出代價的唯一理由。如果你跳過它們，一個由 Auth0 為*不同* Audience 簽名的令牌會通過，惡意的 MCP 伺服器就能對你的伺服器重放令牌。
3. **401 上的 `WWW-Authenticate` 標頭。** 這是 claude.ai 回溯到你的受保護資源元資料的線索。沒有它，claude.ai 看到一個裸露的 401 就放棄了。`resource_metadata` 參數在 MCP 授權 RFC 裡有規範，是啟動重新探索的鉤子。

就這樣。在你的 MCP 路由上註冊中間件，部署，claude.ai 的「連線」按鈕現在就會走完完整的 OAuth2 流程，存入每個使用者的令牌，並帶著真實的使用者身份呼叫你的工具。

> **如果你使用 [`@casys/mcp-server`](https://jsr.io/@casys/mcp-server)：** 以上兩個端點都是內建的。Auth0 OIDC 預設會自動產生受保護資源元資料並配置 JWT 驗證——上面約 80 行程式碼會縮減為一個 YAML 設定區塊。這裡展示的手動方式，就是框架底層做的事情。

## 我踩過的失敗情境（讓你不用重蹈覆轍）

大致按痛苦程度排序：

**「Invalid audience」**——`aud` 聲明不相符。檢查 Auth0 裡你的 API 識別符的尾部斜線。這讓我踩坑了兩次。

**「Missing claims」**——你的 API 的 JWT Profile 還在 Auth0 的舊版預設上。在 **APIs → 你的 API → Settings → Access Token Settings** 裡把它切換到 `RFC 9068`。

**`/authorize` 上的「invalid_request」**——Resource Parameter Compatibility Profile 是關閉的。翻轉那個開關。

**「invalid_redirect_uri」**——Auth0 用戶端裡的 Callback URL 是錯的。它必須一字不差地是 `https://claude.ai/api/mcp/auth_callback`。

**「OAuth2 成功後的 401」**——你的 JWT 中間件在拒絕有效的令牌。通常是因為你忘記固定 `issuer` 和 `audience`，而 `jose` 預設了錯誤的東西。或者因為你的伺服器時鐘偏移超過了 `exp`（Deno Deploy 把這個處理得很好，容器有時候不行）。

**「一次成功，刷新後失敗」**——你在 Auth0 用戶端上停用了 Refresh Token grant。重新啟用它。

**「歐洲可用，美國失敗」**——你的租戶是 `casys.eu.auth0.com`，但有使用者透過 claude.ai 從一個無法路由到它的區域連線。實際上通常不是問題，但如果你在服務高流量，考慮一個美國租戶的映射。

這些都不奇特。所有這些都可以在日誌裡找到。唯一讓它們痛苦的是日誌存在三個不同地方（Auth0 日誌、claude.ai 網路分頁、你的伺服器），所以你有一半的除錯時間花在搞清楚到底是哪個參與者拒絕了請求。

## 你在另一端得到的東西

一旦它運作了，你的 MCP 伺服器會有這些改變：

- **每個請求都有使用者身份。** `payload.sub` 是一個 Auth0 使用者 ID。你可以按使用者限定資料庫查詢、寫入每個使用者的稽核日誌、套用每個使用者的速率限制，所有這些都不需要發明任何東西。
- **多租戶是免費的。** 兩家公司可以從同一個 claude.ai 連線同一個 MCP 伺服器，他們永遠看不到彼此的資料，因為他們的令牌是不同的。
- **撤銷是儀表板上的一次點擊。** 停用 Auth0 使用者，他們下次刷新就失敗，他們就出局了。
- **範圍實際上有作用。** 為你的工具標註所需的範圍（`mcp:read`、`mcp:write`、`mcp:admin`），在中間件裡檢查它們，使用者在同意時就能看到他們在授予哪些能力。
- **你終於可以公開它了。** 帶靜態驗證的公開 MCP 伺服器是等待爆發的意外。帶 OAuth2 的公開 MCP 伺服器是真正的產品。我永遠不會把 mcp-einvoice 放在「貼上這個令牌」的表單後面。有了 OAuth2，我可以把 URL 分享在登陸頁面上，並信任驗證層。

走到這裡花了一個週末。*不*走到這裡的代價——在我自己的伺服器裡重新發明每個使用者的驗證，或者堅持使用靜態金鑰並假裝多租戶之後會自己解決——會是幾個月，而且會交付出錯誤的東西。

## 更大的教訓

這和[行動端 UX 那篇文章](/blog/mcp-apps-mobile-ux-patterns)是同一個教訓，從不同角度看。當用戶端是別人的應用程式時，你沒辦法主導互動模式。claude.ai 決定了 MCP 伺服器必須使用 OAuth2，這個決定把一大堆好的實踐下推給了可能原本不會實作它們的伺服器作者。

我曾經認為那是設門檻。現在我認為那是生態系保持健康的方式。網路之所以在不受信任的程式碼中存活下來，是因為瀏覽器對 Origins 和 CORS 變得積極。MCP 之所以能在不受信任的工具中存活，是因為用戶端對驗證和同意變得積極。如果你在這個領域建構東西，順勢而為——不要交付一個靜態金鑰的 MCP 伺服器然後希望使用者能忍受那種摩擦。設定好 Auth0 一次，複製中間件，你就永遠完成了。

下次你盯著 claude.ai 連線對話框裡的「伺服器 URL」，想知道令牌欄位去哪了：它沒有消失。它是一個特性。
