自从 Anthropic 在 2024 年底发布 Model Context Protocol，**MCP 服务器**就成了人人挂在嘴边的词。但你真正追问"它到底是什么"的时候，得到的要么是一张满是箭头的营销图，要么是一份 80 页的规范文档。两者都没有回答问题。

以下是简短版本：

> **MCP 服务器是一个小型服务，通过 JSON-RPC 向 AI 智能体暴露工具、数据和提示。** 把它想成你的 LLM 的"USB 驱动"：每个系统（Postgres、GitHub、你的业务 API）一个服务器，任何兼容的客户端——Claude、Cursor、VS Code Copilot、自定义智能体——都能连接，无需编写特定的集成代码。

定义这份契约的 [Model Context Protocol](https://modelcontextprotocol.io)（模型上下文协议）由 Anthropic 于 2024 年 11 月发布，随后于 2025 年 12 月移交给 Linux 基金会旗下的 Agentic AI Foundation。它现在是将智能体接入外部系统的事实标准。

下文将解释 MCP 服务器实际暴露什么、真实交互长什么样、它与 REST API 有何不同，以及如果你想在生产中部署一个，需要提前考虑哪些问题。

## MCP 服务器到底做什么

MCP 服务器通过 **JSON-RPC 2.0** 暴露三类原语。每类原语对连接服务器的 AI 客户端有不同的用途。

### 工具（tools）

工具是可调用的函数。MCP 服务器为每个工具声明名称、描述（模型读取描述来决定何时调用）以及输入的 JSON Schema。当智能体决定使用某个工具时，发送一个 `tools/call` 请求；服务器执行逻辑并返回结果。

具体示例：一个账单 MCP 服务器可能暴露名为 `list_invoices` 的工具，参数为 `{ customer_id: string, limit: number }`。模型读取描述后判断这是处理"显示客户 X 最近 5 张发票"的正确工具，并发起调用。服务器查询数据库并返回结构化 JSON。

工具可以读取或写入。MCP 规范对副作用没有任何限制——工具可以创建记录、发送邮件或触发部署。你编写的描述决定了模型是否会调用它，因此准确性至关重要。

### 资源（resources）

资源是可读数据。每个资源有一个 URI 和 MIME 类型。客户端可以调用 `resources/list` 发现可用资源，然后调用 `resources/read` 获取内容。

具体示例：一个文档 MCP 服务器可能将 `file:///project/README.md` 暴露为 text/markdown 资源。在决定使用哪个工具之前，智能体可以读取 README 来了解项目结构。与工具不同，资源是被动的——它们暴露的是模型可以作为上下文拉取的数据，而不是可以触发的操作。

### 提示（prompts）

提示是可复用的消息模板。服务器可以暴露命名模板，客户端检索后插入对话中。具体示例：一个代码审查服务器可能暴露一个 `/summarize` 提示模板，接受 diff 作为输入并返回结构化摘要格式。提示是当前实现中使用最少的原语，但对于标准化业务工作流向模型的呈现方式很有价值。

## JSON-RPC 具体示例

以下是工具定义在 MCP wire 格式中的样子——即服务器在客户端调用 `tools/list` 时返回的内容：

```json
{
  "name": "list_invoices",
  "description": "列出某个客户的发票。返回最多 50 条按日期倒序排列的结果。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "客户的唯一标识符"
      },
      "limit": {
        "type": "number",
        "minimum": 1,
        "maximum": 50,
        "default": 10
      }
    },
    "required": ["customer_id"]
  }
}
```

当智能体调用它时，交换过程如下：

```json
// Agent → MCP server: tools/call
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "list_invoices",
    "arguments": { "customer_id": "cust_8a3f", "limit": 5 }
  }
}

// MCP server → Agent: response
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"id\":\"inv_001\",\"amount\":1200,\"status\":\"paid\"}, ...]"
      }
    ]
  }
}
```

模型拿到结构化文本内容后可以继续推理——对发票进行汇总、标记逾期发票，或调用链中的下一个工具。

## MCP 服务器 vs REST API：真正的区别

这个问题每次都会被提起："MCP 服务器不就是换了个名字的 REST API 吗？"

**简短回答：不是。REST API 假设调用方已经知道接口。MCP 服务器在连接时向调用方自我描述，让 AI 智能体无需任何预先编写的代码就能发现并调用其能力。**

具体来说：

- **REST API** — 你硬编码 `/api/v1/invoices`、查询参数、认证头。调用方具备完整的先验知识。API 变了，你重新部署客户端。
- **MCP 服务器** — 智能体连接后发送 `tools/list`，收到一个机器可读的目录（名称、描述、输入 JSON Schema），然后从中决定调用什么。发现过程在运行时、每个会话中动态发生。服务器新增一个工具，智能体立刻看到。

MCP 也是位于传输层（stdio 或 HTTP）之上的协议层，而**不是**对现有 API 的替代。大多数 MCP 服务器都是适配器：GitHub MCP 服务器在内部调用 GitHub REST API，你的账单 MCP 服务器调用你的账单 API。MCP 是面向智能体的接口；底层数据访问保持原样不变。

实际好处：一个 MCP 服务器，任意 AI 客户端，每个提供商无需自定义集成代码。

## 谁在编写 MCP 服务器

Anthropic 在[官方 MCP 服务器仓库](https://github.com/modelcontextprotocol/servers)中发布了常见系统的参考服务器——文件系统、git、GitHub、PostgreSQL。这些涵盖了最常见的开发者需求，并作为实现参考。

社区已经构建了数百个更多的服务器。[Smithery](https://smithery.ai) 和 [Glama](https://glama.ai/mcp/servers) 是两个目录，收录了可用的 MCP 服务器，涵盖数据库、云服务商、效率工具、开发者服务和特定领域集成。GitHub 上的 `awesome-mcp-servers` 列表是第三个起点。

企业构建内部 MCP 服务器，让智能体访问内部系统：内部 API、专有数据库、自定义工作流。Casys 就是一个例子——我们构建并维护 MCP 服务器，作为我们自己 AI 智能体平台的组成部分。模式始终如一：将内部系统包装在 MCP 接口之后，组织内的任何智能体都可以调用它。

## 如何在生产中构建 MCP 服务器

让 MCP 服务器响应 `tools/call` 只需半天时间。让它处理生产流量——OAuth2 认证、按客户端限流、可观测性、负载下的优雅关闭——则需要相当长的时间。

这些都不在 MCP 规范里。规范定义了消息格式和三种原语。当客户端每分钟发送 500 个请求时如何处理、如何验证 bearer token、如何输出指标——这些是你自己的问题。

两条路可走：

1. **自己搭建。** [生产指南](/blog/mcp-server-guide)涵盖了所有内容：从零开始的 TypeScript 实现、使用 JWT 和 OAuth2 预设的认证、限流、Prometheus 指标、OpenTelemetry 追踪、Docker 和 systemd 部署。如果你正在构建一个需要应对真实使用的服务器，就从那里开始。
2. **从框架出发。** [`@casys/mcp-server`](/mcp-server) 是我们使用和维护的框架——在 Deno 和 Node.js 上，它原生处理认证、限流、背压和可观测性。你专注于编写工具，框架处理其余的事。

如果你计划向智能体暴露多个 MCP 服务器，还可以看看 [MCP Gateway](/blog/mcp-gateway-guide)——一个统一入口，负责对下游服务器进行多路复用和安全管控。

## FAQ — 关于 MCP 服务器的常见问题

### MCP 服务器和 REST API 有什么区别？

REST API 假设调用方已知道端点。MCP 服务器在连接时通过 `tools/list` *自我描述*，让 AI 智能体无需硬编码集成就能发现并调用其能力。MCP 通常*叠加在*现有 API 之上，而不是取代它。

### 如何创建 MCP 服务器？

先安装官方 SDK（[TypeScript](https://github.com/modelcontextprotocol/typescript-sdk)、[Python](https://github.com/modelcontextprotocol/python-sdk)），用 JSON Schema 定义你的工具，然后以 stdio 或 HTTP 方式启动服务器。生产环境（认证、限流、可观测性）请参考我们的[分步指南](/blog/mcp-server-guide)或使用 [`@casys/mcp-server`](/mcp-server)。

### 用什么语言写 MCP 服务器？

任何能说 JSON-RPC 2.0 的语言都可以。Anthropic 维护 **TypeScript**、**Python**、**Java**、**Kotlin** 和 **C#** 的官方 SDK。社区发布了 Go、Rust 和 Deno 的 SDK。由于协议与传输无关，任何有 HTTP 客户端或 stdio 访问权限的语言都能实现一个 MCP 服务器。

### 谁维护 MCP 协议？

[Anthropic](https://anthropic.com) 于 2024 年 11 月发布 MCP，并于 2025 年 12 月将治理权移交给 Linux 基金会旗下的 **Agentic AI Foundation**。该协议现在是开放的、多厂商的，已被 OpenAI、Google、Microsoft 及几乎所有主流智能体平台采用。

### 最常用的 MCP 服务器有哪些？

Anthropic 方面：`filesystem`、`git`、`github`、`postgres`、`slack`、`puppeteer`。社区方面：Notion、Linear、Stripe、AWS、Google Drive、Sentry——可在 [Smithery](https://smithery.ai) 和 [Glama](https://glama.ai/mcp/servers) 两个目录查看，收录数量达数百个。
