Tout le monde parle de **serveurs MCP** depuis qu'Anthropic a publié le protocole fin 2024. Mais quand vous demandez ce que c'est *concrètement*, vous récupérez soit un schéma marketing avec des flèches, soit une spec de 80 pages. Ni l'un ni l'autre ne répond à la question.

Voici la version courte :

> **Un serveur MCP est un petit service qui expose des outils, des données et des prompts à un agent IA, via JSON-RPC.** Pensez "pilote USB" pour vos LLM : un serveur par système (Postgres, GitHub, votre API métier), et n'importe quel client compatible — Claude, Cursor, VS Code Copilot, un agent maison — peut s'y connecter sans code d'intégration spécifique.

Le [Model Context Protocol](https://modelcontextprotocol.io) qui définit ce contrat a été publié par Anthropic en novembre 2024, puis cédé à l'Agentic AI Foundation sous la Linux Foundation en décembre 2025. C'est aujourd'hui le standard de fait pour brancher un agent à un système externe.

La suite explique ce que le serveur expose vraiment, à quoi ressemble un échange réel, en quoi ce n'est pas une API REST, et ce qu'il faut prévoir si vous voulez en mettre un en production.

## Ce que fait concrètement un serveur MCP

Les serveurs MCP exposent trois types de primitives via **JSON-RPC 2.0**. Chacune sert un objectif différent pour le client IA qui se connecte.

### Les tools (outils)

Les tools sont des fonctions appelables. Un serveur MCP déclare chaque outil avec un nom, une description (que le modèle lit pour décider quand l'invoquer) et un JSON Schema pour ses paramètres. Quand l'agent décide d'utiliser un outil, il envoie une requête `tools/call` ; le serveur exécute la logique et renvoie un résultat.

Exemple concret : un serveur MCP de facturation pourrait exposer un outil `list_invoices` avec les paramètres `{ customer_id: string, limit: number }`. Le modèle lit la description, juge que c'est le bon outil pour "montre-moi les 5 dernières factures du client X", et l'appelle. Le serveur interroge la base de données et renvoie du JSON structuré.

Les tools peuvent lire ou écrire. Le spec MCP n'impose aucune restriction sur les effets de bord — un outil peut créer des enregistrements, envoyer des emails ou déclencher un déploiement. La description que vous rédigez guide la décision du modèle de l'invoquer, donc la précision compte.

### Les resources (ressources)

Les resources sont des données lisibles. Chaque ressource a une URI et un type MIME. Un client peut appeler `resources/list` pour découvrir ce qui est disponible, puis `resources/read` pour en lire le contenu.

Exemple concret : un serveur MCP de documentation pourrait exposer `file:///project/README.md` comme ressource text/markdown. Avant de décider quel outil utiliser, l'agent peut lire le README pour comprendre la structure du projet. Contrairement aux tools, les resources sont passives — elles exposent des données que le modèle peut intégrer comme contexte, pas des actions à déclencher.

### Les prompts

Les prompts sont des templates de messages réutilisables. Un serveur peut exposer des templates nommés que le client récupère et insère dans une conversation. Exemple concret : un serveur de revue de code pourrait exposer un template `/summarize` qui prend un diff en entrée et renvoie un format de résumé structuré. Les prompts sont la primitive la moins utilisée dans les implémentations actuelles, mais utiles pour standardiser la façon dont les workflows métier sont présentés aux modèles.

## Exemple concret : un serveur MCP en JSON-RPC

Voici à quoi ressemble une définition d'outil dans le format wire MCP — ce que le serveur renvoie quand le client appelle `tools/list` :

```json
{
  "name": "list_invoices",
  "description": "Liste les factures d'un client. Renvoie jusqu'à 50 résultats triés par date décroissante.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "L'identifiant unique du client"
      },
      "limit": {
        "type": "number",
        "minimum": 1,
        "maximum": 50,
        "default": 10
      }
    },
    "required": ["customer_id"]
  }
}
```

Quand l'agent l'appelle, l'échange ressemble à ceci :

```json
// Agent → serveur MCP : tools/call
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "list_invoices",
    "arguments": { "customer_id": "cust_8a3f", "limit": 5 }
  }
}

// Serveur MCP → Agent : réponse
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"id\":\"inv_001\",\"amount\":1200,\"status\":\"paid\"}, ...]"
      }
    ]
  }
}
```

Le modèle récupère du contenu texte structuré et peut continuer à raisonner à partir de là — résumer les factures, signaler les retards, ou appeler l'outil suivant dans une chaîne.

## Serveur MCP vs API REST : la vraie différence

La question revient à chaque conversation : « un serveur MCP, c'est pas juste une API REST avec un autre nom ? »

**Réponse courte : non. Une API REST suppose un appelant qui connaît déjà l'interface. Un serveur MCP la décrit lui-même au moment de la connexion, pour qu'un agent IA puisse la découvrir sans code écrit à l'avance.**

Concrètement :

- **API REST** — vous codez en dur `/api/v1/invoices`, les query params, les headers d'auth. L'appelant a une connaissance préalable complète. Si l'API change, vous redéployez le client.
- **Serveur MCP** — l'agent se connecte, envoie `tools/list`, reçoit un catalogue lisible par machine (noms, descriptions, schémas JSON d'entrée), et décide à partir de là quoi appeler. La découverte se fait à l'exécution, à chaque session. Si le serveur ajoute un outil, l'agent le voit immédiatement.

MCP est aussi une couche de protocole par-dessus un transport (stdio ou HTTP), **pas** un remplacement de votre API existante. La plupart des serveurs MCP sont des adaptateurs : le serveur MCP GitHub appelle l'API REST GitHub en interne, votre serveur MCP de facturation appelle votre API de facturation. MCP est l'interface côté agent ; l'accès aux données sous-jacentes reste exactement comme il est.

L'avantage concret : un seul serveur MCP, n'importe quel client IA, zéro code d'intégration spécifique par fournisseur.

## Qui écrit des serveurs MCP et pourquoi

Anthropic publie des serveurs de référence pour les systèmes courants — filesystem, git, GitHub, PostgreSQL — dans le [dépôt officiel MCP servers](https://github.com/modelcontextprotocol/servers). Ces serveurs couvrent les besoins développeurs les plus fréquents et servent de références d'implémentation.

La communauté en a construit des centaines d'autres. [Smithery](https://smithery.ai) et [Glama](https://glama.ai/mcp/servers) sont deux registres qui cataloguent les serveurs MCP disponibles, couvrant bases de données, cloud providers, outils de productivité, services développeurs et intégrations métier. La liste `awesome-mcp-servers` sur GitHub est un troisième point de départ.

Les entreprises construisent des serveurs MCP internes pour donner à leurs agents accès à leurs systèmes maison : APIs internes, bases de données propriétaires, workflows personnalisés. Casys en est un exemple — nous construisons et maintenons des serveurs MCP dans le cadre de la plateforme que nous utilisons pour nos propres agents IA. Le schéma est toujours le même : envelopper le système interne derrière une interface MCP, et n'importe quel agent de l'organisation peut l'appeler.

## Comment créer un serveur MCP en production

Faire répondre un serveur MCP à `tools/call` prend un après-midi. Le faire tenir sous du trafic réel — authentification OAuth2, rate limiting par client, observabilité, arrêt propre sous charge — prend bien plus longtemps.

Rien de tout ça n'est dans le spec MCP. Le spec définit le format des messages et les trois primitives. Ce qu'il faut faire quand un client envoie 500 requêtes par minute, comment vérifier un bearer token, comment émettre des métriques — c'est votre problème.

Deux chemins possibles :

1. **Câbler vous-même.** Le [guide de production](/blog/mcp-server-guide) couvre tout : implémentation TypeScript de zéro, auth avec JWT et presets OAuth2, rate limiting, métriques Prometheus, traces OpenTelemetry, déploiement Docker et systemd. Si vous construisez un serveur qui doit survivre à un usage réel, commencez là.
2. **Partir d'un framework.** [`@casys/mcp-server`](/mcp-server) est le framework que nous utilisons et maintenons — il gère l'auth, le rate limiting, la backpressure et l'observabilité nativement, sur Deno comme sur Node.js. Vous écrivez vos outils, le framework s'occupe du reste.

Si vous prévoyez d'exposer plusieurs serveurs MCP à vos agents, regardez aussi le [MCP Gateway](/blog/mcp-gateway-guide) — un seul point d'entrée qui multiplexe et sécurise vos serveurs en aval.

## FAQ — Questions fréquentes sur les serveurs MCP

### Quelle est la différence entre un serveur MCP et une API REST ?

Une API REST suppose que l'appelant connaît déjà les endpoints. Un serveur MCP se *décrit lui-même* au moment de la connexion via `tools/list`, pour qu'un agent IA puisse découvrir et invoquer ses capacités sans intégration codée en dur. MCP s'utilise généralement *par-dessus* une API existante, pas à la place.

### Comment créer un serveur MCP ?

Commencez par installer un SDK officiel ([TypeScript](https://github.com/modelcontextprotocol/typescript-sdk), [Python](https://github.com/modelcontextprotocol/python-sdk)), définissez vos `tools` avec un JSON Schema d'entrée, et lancez le serveur en stdio ou HTTP. Pour la production (auth, rate limiting, observabilité), suivez notre [guide pas-à-pas](/blog/mcp-server-guide) ou utilisez [`@casys/mcp-server`](/mcp-server).

### En quel langage écrire un serveur MCP ?

N'importe lequel qui sait parler JSON-RPC 2.0. Anthropic maintient des SDK officiels en **TypeScript**, **Python**, **Java**, **Kotlin** et **C#**. La communauté a publié des SDK Go, Rust et Deno. Le protocole étant transport-agnostique, tout langage avec un client HTTP ou un accès stdio peut implémenter un serveur.

### Qui maintient le protocole MCP ?

[Anthropic](https://anthropic.com) a publié MCP en novembre 2024 et a transféré la gouvernance à l'**Agentic AI Foundation** sous la **Linux Foundation** en décembre 2025. Le protocole est désormais ouvert, multi-vendor, et adopté par OpenAI, Google, Microsoft et la quasi-totalité des éditeurs d'agents.

### Quels sont les serveurs MCP les plus utilisés ?

Côté Anthropic : `filesystem`, `git`, `github`, `postgres`, `slack`, `puppeteer`. Côté communauté : Notion, Linear, Stripe, AWS, Google Drive, Sentry — voir les registres [Smithery](https://smithery.ai) et [Glama](https://glama.ai/mcp/servers) qui en cataloguent plusieurs centaines.
