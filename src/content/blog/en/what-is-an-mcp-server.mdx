Everyone has been talking about **MCP servers** since Anthropic published the protocol in late 2024. But when you ask what one actually *is*, you get either a marketing diagram with arrows, or an 80-page spec. Neither answers the question.

Here is the short version:

> **An MCP server is a small service that exposes tools, data, and prompts to an AI agent over JSON-RPC.** Think of it as a driver layer for your LLMs: one server per system (Postgres, GitHub, your internal API), and any compatible client — Claude, Cursor, VS Code Copilot, a custom agent — can connect to it without writing provider-specific integration code.

The [Model Context Protocol](https://modelcontextprotocol.io) that defines this contract was published by Anthropic in November 2024, then transferred to the Agentic AI Foundation under the Linux Foundation in December 2025. It is now the de facto standard for connecting an agent to an external system.

The rest of this guide covers what the server actually exposes, what a real exchange looks like on the wire, how it differs from a REST API, and what you need to plan for if you want to run one in production.

## What an MCP Server Actually Does

MCP servers expose three types of primitives over **JSON-RPC 2.0**. Each one serves a different purpose for the AI client connecting to the server.

### Tools

Tools are callable functions. An MCP server declares each tool with a name, a description (which the model reads to decide when to invoke it), and a JSON Schema for its parameters. When the agent decides to use a tool, it sends a `tools/call` request; the server executes the logic and returns a result.

A concrete example: a billing MCP server might expose a tool called `list_invoices` with parameters `{ customer_id: string, limit: number }`. The model reads the description, decides this is the right tool for "show me the last 5 invoices for customer X", and calls it. The server queries the database and returns structured JSON.

Tools can read or write. The MCP spec places no restriction on side effects — a tool can create records, send emails, or trigger a deployment. The description you write is what guides the model's decision to call it, so accuracy matters.

### Resources

Resources are readable data. Each resource has a URI and a MIME type. A client can call `resources/list` to discover what is available, then `resources/read` to fetch content.

A concrete example: a documentation MCP server might expose `file:///project/README.md` as a text/markdown resource. Before deciding which tool to call, the agent can read the README to understand the project structure. Unlike tools, resources are passive — they expose data the model can pull in as context, not actions it can trigger.

### Prompts

Prompts are reusable message templates. A server can expose named templates that the client retrieves and inserts into a conversation. A concrete example: a code review server might expose a `/summarize` prompt template that takes a diff as input and returns a structured summary format. Prompts are the least-used MCP primitive in current implementations, but they are useful for standardizing how domain-specific workflows are presented to models.

## Concrete Example: an MCP Server in JSON-RPC

Here is what a tool definition looks like in the MCP wire format — what the server returns when the client calls `tools/list`:

```json
{
  "name": "list_invoices",
  "description": "List invoices for a customer. Returns up to 50 results sorted by date descending.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "customer_id": {
        "type": "string",
        "description": "The customer's unique identifier"
      },
      "limit": {
        "type": "number",
        "minimum": 1,
        "maximum": 50,
        "default": 10
      }
    },
    "required": ["customer_id"]
  }
}
```

When the agent calls it, the exchange looks like this:

```json
// Agent → MCP server: tools/call
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "list_invoices",
    "arguments": { "customer_id": "cust_8a3f", "limit": 5 }
  }
}

// MCP server → Agent: response
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"id\":\"inv_001\",\"amount\":1200,\"status\":\"paid\"}, ...]"
      }
    ]
  }
}
```

The model gets back structured text content and can continue reasoning from there — summarizing the invoices, flagging overdue ones, or calling the next tool in a chain.

## MCP Server vs REST API: the Real Difference

This comes up in every conversation: "isn't an MCP server just a REST API with a different name?"

**Short answer: no. A REST API assumes a caller who already knows the interface. An MCP server describes itself at connection time, so an AI agent can discover it at runtime without any hardcoded integration code.**

Concretely:

- **REST API** — you hardcode `/api/v1/invoices`, the query parameters, the auth headers. The caller has complete prior knowledge. If the API changes, you redeploy the client.
- **MCP server** — the agent connects, sends `tools/list`, gets back a machine-readable catalog (names, descriptions, input JSON schemas), and decides from there what to call. Discovery happens at runtime, every session. If the server adds a tool, the agent sees it immediately.

MCP is also a protocol layer on top of a transport (stdio or HTTP), **not** a replacement for your existing API. Most MCP servers are adapters: the GitHub MCP server calls the GitHub REST API internally; your billing MCP server calls your billing API. MCP is the AI-facing interface; the underlying data access stays exactly as it is.

The practical upside: one MCP server, any AI client, no custom integration code per provider.

## Who Writes MCP Servers and Why

Anthropic publishes reference servers for common systems — filesystem, git, GitHub, PostgreSQL — in the [official MCP servers repository](https://github.com/modelcontextprotocol/servers). These cover the most common developer needs and serve as implementation references.

The community has built hundreds more. [Smithery](https://smithery.ai) and [Glama](https://glama.ai/mcp/servers) are two registries that catalog available MCP servers, covering databases, cloud providers, productivity tools, developer services, and domain-specific integrations. GitHub's `awesome-mcp-servers` list is a third starting point.

Companies build internal MCP servers to give their agents access to in-house systems: internal APIs, proprietary databases, custom workflows. Casys is one example — we build and maintain MCP servers as part of the platform we use for our own AI agents. The pattern is always the same: wrap the internal system behind an MCP interface, and any agent in the organization can call it.

## How to Build an MCP Server in Production

Getting an MCP server to respond to `tools/call` takes an afternoon. Getting it to handle real traffic — OAuth2 authentication, per-client rate limiting, observability, graceful shutdown under load — takes considerably longer.

None of that is in the MCP spec. The spec defines the message format and the three primitives. What to do when a client sends 500 requests per minute, how to verify a bearer token, how to emit metrics — that is your problem.

Two paths forward:

1. **Wire it yourself.** The [production guide](/blog/mcp-server-guide) covers everything: TypeScript implementation from scratch, auth with JWT and OAuth2 presets, rate limiting, Prometheus metrics, OpenTelemetry traces, Docker and systemd deployment. If you are building a server that needs to survive real usage, start there.
2. **Start from a framework.** [`@casys/mcp-server`](/mcp-server) is the framework we use and maintain — it handles auth, rate limiting, backpressure, and observability out of the box, on both Deno and Node.js. You write your tools; the framework handles the rest.

If you plan to expose multiple MCP servers to your agents, also look at the [MCP Gateway](/blog/mcp-gateway-guide) — a single entry point that multiplexes and secures your downstream servers.

## FAQ — Common Questions About MCP Servers

### How does an MCP server work?

An MCP server speaks JSON-RPC 2.0 over stdio or HTTP. When a client connects, it sends `tools/list` to discover available tools, `resources/list` to discover readable data, and `tools/call` to invoke a specific tool. The server handles each request and returns structured content the agent can reason over. Discovery happens at runtime — the agent does not need prior knowledge of what the server exposes.

### What is the difference between MCP and a REST API?

A REST API is designed for callers who already know what endpoints exist and what parameters to pass. An MCP server *describes itself* at connection time via `tools/list`, so an AI agent can discover and invoke its capabilities without any hardcoded integration. MCP is typically used *on top of* an existing API, not instead of one — most MCP servers are adapters that call a REST API internally.

### How do I build an MCP server?

Start by installing an official SDK ([TypeScript](https://github.com/modelcontextprotocol/typescript-sdk), [Python](https://github.com/modelcontextprotocol/python-sdk)), define your `tools` with a JSON Schema for inputs, and run the server over stdio or HTTP. For production (auth, rate limiting, observability), follow our [step-by-step guide](/blog/mcp-server-guide) or use [`@casys/mcp-server`](/mcp-server).

### What languages can I use to write an MCP server?

Any language that can speak JSON-RPC 2.0. Anthropic maintains official SDKs in **TypeScript**, **Python**, **Java**, **Kotlin**, and **C#**. The community has published SDKs for Go, Rust, and Deno. Because the protocol is transport-agnostic, any language with an HTTP client or stdio access can implement a server.

### What are the most popular MCP servers?

From Anthropic: `filesystem`, `git`, `github`, `postgres`, `slack`, `puppeteer`. From the community: Notion, Linear, Stripe, AWS, Google Drive, Sentry — see the [Smithery](https://smithery.ai) and [Glama](https://glama.ai/mcp/servers) registries, which catalog several hundred servers.
