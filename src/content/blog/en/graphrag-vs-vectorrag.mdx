## The Real Problem with Semantic Search

Every RAG tutorial starts the same way: embed your documents, embed the query, find the nearest vectors, stuff the chunks into the context window. It works surprisingly well for simple questions. For anything that requires reasoning across **relationships**, it breaks in ways that are hard to debug and impossible to fix at query time.

The failure mode is not that the retriever returns wrong results. It is that it returns results that are **locally correct but globally misleading**. You ask "what caused the Q3 revenue drop?" and get three chunks about Q3, all accurate, none of which contains the supplier issue from Q2 that set everything off. The relationship exists in your corpus. Vector RAG cannot traverse it.

<ExcalidrawDiagram src="graphrag-vs-vectorrag/vector-vs-graph-same-question" alt="Same question, two strategies: Vector RAG retrieves similar chunks and misses the root cause; GraphRAG traverses the relationship chain and finds it" />

GraphRAG was built to solve exactly this. But the literature and vendors often present it as a **replacement** for Vector RAG. This post argues the opposite: they are two **complementary** tools, answering two different query intents, and a good stack runs both together.

---

## Vector RAG, and Why It Works Most of the Time

Vector RAG became the default for a simple reason: **in the majority of document QA cases, semantic similarity is a reasonable proxy for relevance**.

The principle boils down to three steps:

1. **Indexing** — chunk the documents, embed each chunk with a model (OpenAI, Voyage, Cohere, or a local model like bge-small), store the vectors in an index (pgvector, Qdrant, Pinecone, Weaviate, Milvus).
2. **Query** — embed the question, compute cosine similarity between the query vector and chunk vectors.
3. **Synthesis** — take the top-k chunks, pass them as context to an LLM.

This fits in an afternoon of code, scales to millions of documents with real infra, and correctly answers most factual questions. That is why every framework (LangChain, LlamaIndex, Haystack) makes it their base primitive.

Where it breaks down — and no prompt engineering will overcome these constraints:

- **No multi-hop reasoning.** If the answer connects A → B → C and each link lives in a separate chunk, the retriever returns the chunk closest to the query. It does not chain. A systematic evaluation ([arxiv 2502.11371](https://arxiv.org/abs/2502.11371)) shows Vector RAG can score **0%** on some relationship-dependent tasks.
- **Semantic similarity ≠ relevance.** "The deployment failed" and "we rotated the API key yesterday" have low cosine similarity. Their connection is causal, not semantic.
- **No entity resolution.** "MSFT", "Microsoft", and "Microsoft Corporation" are three separate vectors.
- **Query-independent retrieval.** Whether the question is factual or synthetic, you get back the same type of chunk.
- **Chunk boundary fragmentation.** A relationship spanning a paragraph gets split into two orphaned chunks.

None of these are blockers for document QA at reasonable scale. They compound in enterprise environments — large, heterogeneous, internally interconnected corpora. That is where you need something else.

## What GraphRAG Actually Adds

GraphRAG (Graph Retrieval-Augmented Generation) combines a knowledge graph with the RAG retrieval loop. The retrieval layer is no longer a flat vector index: it is a **graph** where nodes are entities (people, concepts, places, organizations) and edges are typed relationships.

The fundamental difference: **retrieval becomes traversal**. You are not looking for chunks similar to the query — you follow paths from an anchor entity to connected context that embedding similarity would never surface.

<ExcalidrawDiagram src="graphrag-vs-vectorrag/graphrag-knowledge-graph" alt="Knowledge graph visualization: typed entities, typed relationships, traversal path highlighted from query anchor to root cause" />

All serious implementations share the same offline sequence:

1. **Chunking** — split the corpus into text units. Standard.
2. **Entity and relationship extraction** — an LLM pass extracts named entities, typed relationships ("acquired", "caused", "mitigated") and key claims. This is the expensive step: $20–50 per million tokens at GPT-4 rates, and the final graph quality depends directly on extraction quality.
3. **Community detection** — a clustering algorithm (Leiden in Microsoft GraphRAG, hierarchical clustering elsewhere) groups entities, each cluster gets a generated summary. These hierarchical summaries enable thematic queries across the entire corpus.
4. **Local or global query** — **local search** traverses entity neighborhoods for precise facts; **global search** aggregates community summaries for thematic synthesis.

### A Broader Ecosystem Than You Might Think

The name that circulates most is [Microsoft GraphRAG](https://github.com/microsoft/graphrag), because Microsoft open-sourced it with solid marketing. It is neither the only option, nor necessarily the best for your use case:

- **[LightRAG](https://github.com/HKUDS/LightRAG)** (HKU) — lighter open-source, faster indexing, cheaper on tokens.
- **[nano-graphrag](https://github.com/gusye1234/nano-graphrag)** — ~1,000 lines of Python, no Microsoft dependencies, perfect for learning the pattern.
- **[Neo4j + LangChain](https://python.langchain.com/docs/integrations/graphs/neo4j_cypher/)** — native Cypher graph, mature LangChain integration.
- **[LlamaIndex PropertyGraphIndex](https://docs.llamaindex.ai/en/stable/module_guides/indexing/lpg_index_guide/)** — fastest route to a working POC.
- **[FalkorDB](https://www.falkordb.com/)** — Redis-based graph DB designed for RAG/agent pipelines.
- **[Kùzu](https://kuzudb.com/)** — embedded graph DB, columnar + Cypher, zero infrastructure.

Approaches vary on indexing cost, query latency, multilingual support, and incremental update capability. **None is universally better.**

### The Real Costs

Let's be honest about what GraphRAG costs:

- **Build time**: paid LLM extraction, offline clustering from minutes to hours. Any update requires re-indexing (full or partial depending on implementation).
- **Query time**: local search ~200ms, global search several seconds (map-reduce over communities).
- **Quality**: depends on the extractor LLM's ability to normalize entities ("MSFT" = "Microsoft"?) and correctly type relationships. A weak extractor gives you a weak graph.

Where GraphRAG excels:
- Thematic synthesis ("what risk factors appear across these 500 reports?")
- Causal and multi-hop questions ("what caused X?")
- Dense, heterogeneous corpora with explicit relationships (research papers, legal documents, regulatory databases)

Where it disappoints:
- Simple factual lookups (over-engineered)
- Small corpora (indexing cost not justified)
- Real-time updates (batch by design)

The [arxiv 2506.05690 paper](https://arxiv.org/abs/2506.05690) is explicit: **GraphRAG often underperforms on simple retrieval, and strongly inverts the trend on multi-hop tasks**. The decision point is the nature of the question — not the size of the corpus.

## Head-to-Head

| | **Vector RAG** | **GraphRAG** |
|---|---|---|
| **Data model** | Flat vector index | Typed entity graph |
| **Retrieval operation** | Cosine nearest-neighbour | Traversal + communities |
| **Multi-hop** | No | Yes (by design) |
| **Updates** | Incremental (re-embed doc) | Batch (re-index entities) |
| **Indexing cost** | Low (embed pass) | High (LLM extraction) |
| **Query latency** | ~10–50 ms | Local: ~200 ms · Global: seconds |
| **Strength** | Fact retrieval, similarity | Thematic synthesis, causal chains |
| **Weakness** | Relational reasoning | Simple factual lookup |

Read diagonally: Vector RAG wins on simple, GraphRAG wins on relational, neither wins both. That asymmetry is precisely what makes the next section interesting.

## The Real Production Pattern: Hybrid

The trick question is not "which one to pick?". It is "**how do you route the query to the right one?**".

<ExcalidrawDiagram src="graphrag-vs-vectorrag/rag-decision-tree" alt="X/Y decision map for choosing between Vector RAG, GraphRAG, Vector streaming, or Hybrid based on query complexity and corpus update frequency" />

Serious stacks do not choose — they combine. Three patterns that work:

**1. Upstream router by query intent.** A small classifier (or a lightweight LLM) determines whether the question is factual-similar or relational-causal, then dispatches to the vector store or the graph. LangChain offers this pattern via its routers. Simple, explicit, measurable independently.

**2. Vector RAG first pass, GraphRAG for enrichment.** The vector retriever surfaces k candidates (fast, cheap), then a graph pass adds connected entities to the retrieved chunks. The LLM synthesizes over both. You pay ~50ms of vector + ~200ms of local graph pass — acceptable in most cases.

**3. Dual indexes on the same content, hybrid scoring.** Each document is indexed twice (embeddings + entity extraction). At query time, both lookups run and scores are merged. More expensive at indexing but very robust — this is the default approach of LlamaIndex's `PropertyGraphIndex`.

The real architectural question is **latency and indexing cost**. Vector RAG can handle a million documents for a few dollars a month. GraphRAG on the same corpus costs thousands to index. Good judgment means reserving GraphRAG for questions where it makes a measurably different — which means monitoring both pipelines in production and comparing by question type.

## Where to Start

Starting from scratch, the order that works:

1. **Vector RAG first**, any framework (pgvector + LangChain or LlamaIndex) on a subset of your corpus. Measure quality on your real questions.
2. **Identify where Vector RAG fails.** If it's "what are the themes" or "what caused X", you have a relational problem — move to step 3. Otherwise, stay with Vector RAG.
3. **Try nano-graphrag or LightRAG** before Microsoft GraphRAG — faster to prototype, cheaper. Measure on the same questions as step 1.
4. **Integrate both via a router.** The router code is thirty lines. The quality gain on relational questions is well worth the cost of one extra call.

```bash
# Minimal example with nano-graphrag
pip install nano-graphrag
```

```python
from nano_graphrag import GraphRAG

graph = GraphRAG(working_dir="./cache")
graph.insert(["doc1 text...", "doc2..."])
print(graph.query("what caused X?"))
```

For a serious corpus, expect two to three days of indexing plus prompt engineering before quality stabilizes. The first run always costs more than expected because of failed entity extractions — that gets fixed by iterating on the extraction prompt.

---

## The Decision in One Sentence

Vector RAG answers "find me what looks like this". GraphRAG answers "find me what is connected to this". If most of your queries are the first type, stay with Vector RAG. If they regularly require the second, **add** a GraphRAG layer — not instead of, but alongside, routed by intent.

The real choice is not binary. It is a **routing** choice.
